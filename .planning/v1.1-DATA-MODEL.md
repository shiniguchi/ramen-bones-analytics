## 1. Architecture Summary

Pragmatic star schema:

```
                    ┌──────────────┐
                    │ dim_customer │  (lifetime attributes, one row per card_hash)
                    └──────┬───────┘
                           │ joined on (restaurant_id, card_hash)
                           ▼
┌──────────────┐   ┌──────────────────────┐   ┌──────────────┐
│ transactions │──▶│   fct_transactions   │──▶│  rollup MVs  │
│  (raw)       │   │  (atomic fact MV)    │   │ (chart-ready)│
└──────────────┘   └──────────────────────┘   └──────────────┘
                           ▲
                  dim_date (derived view, not materialized)
```

**Design decisions (locked):**

1. **Payment/country/sales_type are denormalized on the fact**, not separate dimension tables. Cardinality is low, join cost is non-zero, and 1-tenant scale doesn't justify the normalization.
2. **Two repeater-bucket columns are materialized on `fct_transactions`:** `lifetime_bucket` ("how this customer ended up") and `visit_seq_bucket` ("point-in-time bucket at the moment of that visit"). Charts pick whichever matches their question — no second revision needed.
3. **Rollup MVs materialize at day grain**; week/month views re-bucket in the wrapper views via `date_trunc`. One MV serves three granularities.
4. **Avg ticket is NOT materialized** in `mv_repeater_daily` — computed on read as `revenue_cents / NULLIF(visits, 0)`. Avoids a non-additive measure in the MV.
5. **Dropdown filter options (payment method, country) are auto-populated** via `SELECT DISTINCT` at page load. No hardcoded whitelist. Adding a new country = zero code change.
6. **All refresh stays inside the existing `refresh_analytics_mvs()`** function. No new cron, no new infra. DAG-ordered.
7. **All new MVs use `REFRESH MATERIALIZED VIEW CONCURRENTLY`** — unique index mandatory. This is already a project-wide rule.
8. **Every read path goes through a `*_v` security-invoker wrapper** with the JWT `restaurant_id` filter. Raw MVs have `REVOKE ALL` on `authenticated`. Same pattern as Phases 1–3.
9. **Forkability/public-flip (v1.0 Plan 05-06 Task 2) is deferred** — repo stays private until onboarding other restaurants becomes an active goal.

## 2. Base Table Mutation — `transactions`

Phase 07 adds two columns to the existing `public.transactions` table.

### 2.1 Migration: `0018_transactions_country_cardtype.sql`

```sql
ALTER TABLE public.transactions
  ADD COLUMN wl_issuing_country char(2),
  ADD COLUMN card_type          text;

COMMENT ON COLUMN public.transactions.wl_issuing_country IS
  'ISO-2 country code of the payment card issuer (from Worldline). NULL for cash. Promoted from stg_orderbird_order_items.wl_issuing_country, first row per invoice.';
COMMENT ON COLUMN public.transactions.card_type IS
  'Payment card brand (Visa / MasterCard / Maestro / etc). NULL for cash. Promoted from stg_orderbird_order_items.card_type, first row per invoice.';
```

### 2.2 One-shot backfill SQL

```sql
UPDATE public.transactions t
SET wl_issuing_country = s.wl_issuing_country,
    card_type          = s.card_type
FROM (
  SELECT DISTINCT ON (restaurant_id, invoice_number)
         restaurant_id, invoice_number, wl_issuing_country, card_type
  FROM public.stg_orderbird_order_items
  ORDER BY restaurant_id, invoice_number, row_index
) s
WHERE t.restaurant_id = s.restaurant_id
  AND t.source_tx_id  = s.invoice_number;
```

### 2.3 New index (supports window fns in Phase 08)

```sql
CREATE INDEX transactions_restaurant_card_time
  ON public.transactions (restaurant_id, card_hash, occurred_at)
  WHERE card_hash IS NOT NULL;
```

Partial index — cash rows (`card_hash IS NULL`) never participate in window-function queries, so excluding them cuts index size ~40%.

### 2.4 Loader update

CSV loader (see Phase 2 code) must write `wl_issuing_country` and `card_type` on future ingests. Idempotency preserved via the existing `(restaurant_id, source_tx_id)` natural-key upsert.

## 3. `dim_customer` — Lifetime Customer Dimension (MV)

**Phase:** 08
**Grain:** one row per `(restaurant_id, card_hash)`. Cash customers (NULL card_hash) are **not** in this table.
**Refresh:** nightly, `REFRESH MATERIALIZED VIEW CONCURRENTLY`.

### 3.1 Columns

| Column | Type | SQL logic |
|---|---|---|
| `restaurant_id` | uuid | passthrough |
| `card_hash` | text | passthrough (NOT NULL; cash excluded) |
| `first_visit_at` | timestamptz | `MIN(occurred_at)` |
| `last_visit_at` | timestamptz | `MAX(occurred_at)` |
| `first_visit_date` | date | `(MIN(occurred_at) AT TIME ZONE 'Europe/Berlin')::date` |
| `cohort_week` | date | `date_trunc('week', first_visit_date)::date` |
| `cohort_month` | date | `date_trunc('month', first_visit_date)::date` |
| `lifetime_visits` | int | `COUNT(*)` |
| `lifetime_revenue_cents` | bigint | `SUM(gross_cents)` |
| `lifetime_avg_ticket_cents` | int | `(SUM(gross_cents) / COUNT(*))::int` |
| `lifetime_bucket` | text | CASE ladder (see §3.2) |
| `primary_country` | char(2) | `mode() WITHIN GROUP (ORDER BY wl_issuing_country)` — most-used country |

### 3.2 `lifetime_bucket` CASE ladder

```sql
CASE
  WHEN lifetime_visits = 1                    THEN 'first_timer'
  WHEN lifetime_visits = 2                    THEN '2x'
  WHEN lifetime_visits = 3                    THEN '3x'
  WHEN lifetime_visits BETWEEN 4 AND 5        THEN '4-5x'
  ELSE                                              '6+'
END
```

Bucket boundaries may be re-tuned in Phase 08 after eyeballing the real visit-count distribution — but any change must stay in this one CASE expression and nowhere else.

### 3.3 Indexes

```sql
CREATE UNIQUE INDEX dim_customer_pk
  ON public.dim_customer (restaurant_id, card_hash);  -- required for CONCURRENTLY

CREATE INDEX dim_customer_cohort_month
  ON public.dim_customer (restaurant_id, cohort_month);

CREATE INDEX dim_customer_bucket
  ON public.dim_customer (restaurant_id, lifetime_bucket);
```

### 3.4 RLS wrapper

```sql
CREATE VIEW public.dim_customer_v WITH (security_invoker = true) AS
SELECT * FROM public.dim_customer
WHERE restaurant_id::text = (auth.jwt()->>'restaurant_id');

REVOKE ALL ON public.dim_customer FROM authenticated, anon;
GRANT SELECT ON public.dim_customer_v TO authenticated;
```

## 4. `fct_transactions` — Atomic Fact MV

**Phase:** 08
**Grain:** one row per `(restaurant_id, source_tx_id)` — matches `transactions` grain exactly.
**Refresh:** nightly, `REFRESH MATERIALIZED VIEW CONCURRENTLY`. Depends on `dim_customer` (DAG step 2).

This is **the hot table**. Every chart query reads from here (via `fct_transactions_v`). Every filter dropdown filters here. The whole point of the star-schema refactor is that this one table answers all analytical questions without re-running window functions per chart.

### 4.1 Columns (30 total)

| # | Column | Type | SQL logic / source |
|---|---|---|---|
| **— identity —** | | | |
| 1 | `restaurant_id` | uuid | passthrough from `transactions` |
| 2 | `source_tx_id` | text | passthrough |

### 4.2 `visit_seq_bucket` CASE ladder

```sql
CASE
  WHEN card_hash IS NULL                    THEN 'cash_anonymous'
  WHEN visit_seq = 1                        THEN 'first_timer'
  WHEN visit_seq = 2                        THEN '2x'
  WHEN visit_seq = 3                        THEN '3x'
  WHEN visit_seq BETWEEN 4 AND 5            THEN '4-5x'
  ELSE                                            '6+'
END
```

**Note:** `lifetime_bucket` uses the same labels but a different source (`dim_customer.lifetime_visits`). A customer with 7 lifetime visits always has `lifetime_bucket = '6+'` on every one of their historical rows; their `visit_seq_bucket` progresses from `first_timer` → `2x` → `3x` → `4-5x` → `6+` across those rows.

### 4.3 Full build SQL

```sql
CREATE MATERIALIZED VIEW public.fct_transactions AS
WITH tx AS (
  SELECT
    t.restaurant_id,
    t.source_tx_id,
    t.card_hash,
    t.occurred_at,
    t.gross_cents,
    t.net_cents,
    t.tip_cents,
    t.sales_type,
    t.payment_method,
    t.wl_issuing_country,
    t.card_type,
    (t.occurred_at AT TIME ZONE 'Europe/Berlin')::date AS business_date,
    ROW_NUMBER() OVER w AS visit_seq,
    LAG(t.occurred_at) OVER w AS prev_visit_at
  FROM public.transactions t
  WINDOW w AS (PARTITION BY t.restaurant_id, t.card_hash ORDER BY t.occurred_at)
)
SELECT
  tx.restaurant_id,
  tx.source_tx_id,
  tx.card_hash,
  tx.occurred_at,
  tx.business_date,
  date_trunc('week',  tx.business_date)::date  AS business_week,
  date_trunc('month', tx.business_date)::date  AS business_month,
  EXTRACT(HOUR   FROM tx.occurred_at AT TIME ZONE 'Europe/Berlin')::smallint AS hour_of_day,
  EXTRACT(ISODOW FROM tx.business_date)::smallint AS day_of_week,
  (EXTRACT(ISODOW FROM tx.business_date) IN (6, 7)) AS is_weekend,
  tx.gross_cents,
  tx.net_cents,
  tx.tip_cents,
  tx.sales_type,
  tx.payment_method,
  tx.wl_issuing_country,
  (tx.wl_issuing_country = 'DE') AS is_domestic,
  tx.card_type,
  tx.visit_seq,
  (tx.visit_seq = 1) AS is_first_visit,
  tx.prev_visit_at,
  (tx.business_date - (tx.prev_visit_at AT TIME ZONE 'Europe/Berlin')::date) AS days_since_prev_visit,
  dc.first_visit_date,
  dc.cohort_week,
  dc.cohort_month,
  (tx.business_date - dc.first_visit_date) AS days_since_first_visit,
  COALESCE(dc.lifetime_bucket, 'cash_anonymous') AS lifetime_bucket,
  CASE
    WHEN tx.card_hash IS NULL         THEN 'cash_anonymous'
    WHEN tx.visit_seq = 1             THEN 'first_timer'
    WHEN tx.visit_seq = 2             THEN '2x'
    WHEN tx.visit_seq = 3             THEN '3x'
    WHEN tx.visit_seq BETWEEN 4 AND 5 THEN '4-5x'
    ELSE                                    '6+'
  END AS visit_seq_bucket
FROM tx
LEFT JOIN public.dim_customer dc
  ON dc.restaurant_id = tx.restaurant_id
 AND dc.card_hash     = tx.card_hash;
```

## 5. Chart Rollup MVs

**Phase:** 09
**Source:** all read from `public.fct_transactions` (NOT `fct_transactions_v` — the refresh runs as owner, not end user).
**Refresh:** nightly, `REFRESH MATERIALIZED VIEW CONCURRENTLY`. DAG step 3, parallel-safe (all depend only on `fct_transactions`).

Each rollup gets a matching `*_v` wrapper following the Phase 3 pattern.

### 5.1 `mv_new_customers_daily` — Chart 1 (new customers per period)

```sql
CREATE MATERIALIZED VIEW public.mv_new_customers_daily AS
SELECT
  restaurant_id,
  first_visit_date AS day,
  COUNT(*)::int   AS new_customers
FROM public.fct_transactions
WHERE is_first_visit
GROUP BY 1, 2;

CREATE UNIQUE INDEX mv_new_customers_daily_pk
  ON public.mv_new_customers_daily (restaurant_id, day);
```

Week/month variants are derived in the wrapper view via `date_trunc`.

### 5.2 `mv_repeater_daily` — Charts 2, 3, 4 (one MV, three charts)

```sql
CREATE MATERIALIZED VIEW public.mv_repeater_daily AS
SELECT
  restaurant_id,
  business_date       AS day,
  lifetime_bucket,
  visit_seq_bucket,
  sales_type,
  payment_method,
  wl_issuing_country,
  COUNT(DISTINCT card_hash) AS customers,
  COUNT(*)                  AS visits,
  SUM(gross_cents)          AS revenue_cents,
  SUM(tip_cents)            AS tip_cents
FROM public.fct_transactions
GROUP BY 1, 2, 3, 4, 5, 6, 7;

CREATE UNIQUE INDEX mv_repeater_daily_pk
  ON public.mv_repeater_daily (
    restaurant_id, day, lifetime_bucket, visit_seq_bucket,
    sales_type, payment_method, wl_issuing_country
  );
```

**Avg ticket is computed on read** in the wrapper view or load function:
```sql
revenue_cents::numeric / NULLIF(visits, 0)
```

### 5.3 `mv_retention_monthly` — Chart 5 monthly variant

Weekly retention already exists in `retention_curve_v` (Phase 3). This adds the monthly variant.

```sql
CREATE MATERIALIZED VIEW public.mv_retention_monthly AS
SELECT
  restaurant_id,
  cohort_month,
  ((business_month - cohort_month) / 30)::int AS period_offset_months,
  COUNT(DISTINCT card_hash) AS retained_customers
FROM public.fct_transactions
WHERE card_hash IS NOT NULL
GROUP BY 1, 2, 3;

CREATE UNIQUE INDEX mv_retention_monthly_pk
  ON public.mv_retention_monthly (restaurant_id, cohort_month, period_offset_months);
```

### 5.4 `mv_inter_visit_histogram` — Chart 6 (return-day histogram)

```sql
CREATE MATERIALIZED VIEW public.mv_inter_visit_histogram AS
SELECT
  restaurant_id,
  business_date AS day,
  CASE
    WHEN days_since_prev_visit BETWEEN 0  AND 3  THEN '0-3d'
    WHEN days_since_prev_visit BETWEEN 4  AND 7  THEN '4-7d'
    WHEN days_since_prev_visit BETWEEN 8  AND 14 THEN '8-14d'
    WHEN days_since_prev_visit BETWEEN 15 AND 30 THEN '15-30d'
    WHEN days_since_prev_visit BETWEEN 31 AND 60 THEN '31-60d'
    ELSE                                                '61+d'
  END AS bucket_label,
  COUNT(*) AS return_visits
FROM public.fct_transactions
WHERE days_since_prev_visit IS NOT NULL
GROUP BY 1, 2, 3;

CREATE UNIQUE INDEX mv_inter_visit_histogram_pk
  ON public.mv_inter_visit_histogram (restaurant_id, day, bucket_label);
```

Bucket boundaries live in this single CASE. To change them: one migration, no UI change.

## 7. Filter Contract (Phase 6)

6 global filters, all exposed as query params on the SvelteKit dashboard route. Every chart honors every filter.

| # | Filter | Query param | Type / values | Maps to WHERE clause on `fct_transactions_v` |
|---|---|---|---|---|
| 1 | Date range | `from`, `to` | ISO dates | `business_date BETWEEN $from AND $to` |
| 2 | Granularity | `grain` | `day` / `week` / `month` | `date_trunc($grain, business_date)` in SELECT, not WHERE |
| 3 | Sales type | `sales` | `all` / `INHOUSE` / `TAKEAWAY` | `sales_type = $sales` (or no-op if `all`) |
| 4 | Payment method | `pay` | `all` / any distinct value | `payment_method = $pay` (or no-op) |
| 5 | Issuing country | `country` | `all` / `DE` / `non-DE` / any ISO-2 | `wl_issuing_country = $country` (or `!= 'DE'` for `non-DE`, or no-op for `all`) |
| 6 | Repeater bucket | `bucket` | `all` / `first_timer` / `2x` / `3x` / `4-5x` / `6+` | `lifetime_bucket = $bucket` (or no-op) |

### 7.1 Validation

All 6 query params are validated by a **single zod schema** in `src/lib/filters.ts`. The SSR load function:
1. Parses `url.searchParams` through the zod schema.
2. Composes a typed `FilterState` object.
3. Passes `FilterState` to query builders that emit parameterized Supabase calls.
4. **Never** interpolates filter values into SQL strings.

A `ci-guards` regex forbids `${` inside `.from(` expressions in `src/`.

### 7.2 Dropdown population (dynamic)

Payment method and country dropdown options are fetched at page load via:
```sql
SELECT DISTINCT payment_method FROM fct_transactions_v ORDER BY payment_method;
SELECT DISTINCT wl_issuing_country FROM fct_transactions_v WHERE wl_issuing_country IS NOT NULL ORDER BY wl_issuing_country;
```

No hardcoded whitelist anywhere. Adding a new country = zero code change.

### 7.3 "All" sentinel

Every dropdown has an "All" option that maps to **omitting the WHERE clause entirely** (not `WHERE col IS NOT NULL` — that would exclude cash rows unintentionally).

## 9. Bug Fixes (Phase 11)

### 9.1 `BUG-01` — NVR card always empty

**Symptom** (from `04-VERIFICATION.md` Gap E): `NewVsReturningCard` shows "No sales recorded in this window" on every range, including `range=all` with 6,842 transactions present.

**Root-cause candidates** (investigate in this order):
1. Loader query in `src/routes/+page.server.ts` passing wrong `range` filter to `new_vs_returning_v`.
2. Empty-state trigger in `NewVsReturningCard.svelte` firing on truthy-but-empty result shape.
3. `new_vs_returning_v` SQL itself returning zero rows for the queried tenant.

**Definition of done:** Non-empty card on `range=all`; the empty state only triggers when the filtered set is genuinely empty.

### 9.2 `BUG-02` — LTV chart only 3 weeks of bars

**Symptom** (from `04-VERIFICATION.md` Gap F): LTV chart on `range=all` shows only 3 bars (2026-03-09 / 03-16 / 03-23) despite the "Based on 10 months of history" caveat.

**Root-cause candidates:**
1. `ltv_mv` is sparse — only materializing rows for recent cohorts.
2. Chart window/truncation in `LtvCard.svelte` clipping older bars.
3. Date filter in the SvelteKit loader applying to `ltv_mv.period_start` instead of cohort-inception date.

**Definition of done:** LTV chart shows full history (10+ months of bars) on `range=all`; root cause fixed at its source, not worked around in the chart.
