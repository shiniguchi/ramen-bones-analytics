## Accomplishments

- **D-17 horizontal-scroll wrapper shipped on both Forecast cards.** Both cards now wrap their `<Chart>` in a `<div bind:this={scrollerRef} bind:clientWidth={cardW} class="mt-4 h-64 overflow-x-auto overscroll-x-contain chart-touch-safe">` exactly mirroring the CalendarRevenueCard:194-263 pattern. `chartW = computeChartWidth(totalSlots, cardW)` flows past + future bucket counts into the canvas-width helper.
- **Scroll-to-today RAF effect shipped on both cards.** First paint lands today at ~60% of the visible viewport (`todayPct = pastBuckets / totalSlots`). Effect uses `scrollWidth < w * 0.9` poll guard (max 30 attempts) to wait for the inner SVG layout to catch up to chartW changes — handles the 1-2 frame lag between chartW set and DOM commit. Re-fires on grain change via chartW reactivity. `lastSetScrollLeft` tracks our own writes so user-scrolling stops auto-positioning but layout-driven width changes don't.
- **Past/future Spline split shipped on both cards.** Replaced the single Spline `{#each}` block (lines 148-159 in both files pre-edit) with two blocks: past (faded solid via `stroke-opacity={0.7}`, `curve={curveMonotoneX}`) + future (dashed via `stroke-dasharray={'4 4'}`, `curve={curveMonotoneX}`). Both branches preserve `stroke-width={isNaive ? 1 : 2}`. CI Area band stays single continuous Area per model with `fillOpacity={0.06}` (D-04). `{#if split.past.length > 0}` and `{#if split.future.length > 0}` guards keep cold-start (no actuals → all-future) and pre-D-15 (no past rows) backward-compat.
- **xDomain UNCHANGED on both Forecast cards** (per RESEARCH.md §16.1-05 explicit guidance). Data-driven `[parseISO(allDates[0]), parseISO(allDates[allDates.length-1])]` already picks up the windowed leftmost `target_date` once 16.1-04 ships rows back to `window_start`. No widening edit needed (different from Calendar* cards which DO need widening).
- **C-02 / C-03 invariants preserved.** `let:data` count = 0 in both files (comment-stripped grep); `tooltipContext={{ mode: 'bisect-x', touchEvents: 'auto' }}` preserved on every Chart; `{#snippet children({ data })}` form preserved.
- **D-16 OUT OF SCOPE preserved.** Tooltip.Root + ForecastHoverPopup delegation untouched on both cards (per plan + RESEARCH.md §16.1-05).
- **B3 typed Spline lambdas preserved.** Both past + future Splines use `x={(r: { d: Date }) => r.d}` and `y={(r: { yhat_mean: number }) => r.yhat_mean}` (no implicit-any regression).

## Decisions Made

- **xDomain stays unchanged on Forecast cards.** Per RESEARCH.md §16.1-05: data-driven `[parseISO(allDates[0]), parseISO(allDates[allDates.length-1])]` already picks up windowed leftmost `target_date` post-D-15 — there are no bars to define a competing anchor. Calendar* cards needed explicit chartXDomain widening (D-03) because their xDomain is anchored on `bucketRange(w.from, ..., grain)` which is independent of forecast rows; Forecast cards have no such anchor.
- **curveMonotoneX preserved on both past + future Spline branches** (per RESEARCH.md §16.1-05 explicit guidance). RevenueForecastCard / InvoiceCountForecastCard pre-existing styling uses curveMonotoneX; preserve on both new branches. Calendar* cards intentionally do NOT use curveMonotoneX on the past/future Splines (they use the LayerChart default). The two card families have intentionally different curve choices — Forecast cards favor smoothed forecast lines, Calendar* cards favor crisp bar+line co-render.
- **D-16 tooltip extension OUT OF SCOPE** per plan + RESEARCH.md §16.1-05. Forecast cards delegate Tooltip.Root to `<ForecastHoverPopup>` which already shows per-model values; the friend-owner's D-16 gripe was about Calendar* cards (where bars + lines coexist). Tooltip.Root snippet form preserved as-is.
- **scrollerRef state shape: `<HTMLDivElement | undefined>()` not `null`** — matches CalendarRevenueCard:286 idiom. Functionally equivalent; null/undefined guard handles both. Used the typed `<HTMLDivElement | undefined>()` form rather than the bare `<HTMLDivElement>()` to make the pre-bind state explicit.
- **lastSetScrollLeft tracked as plain `let lastSetScrollLeft = 0`** (NOT $state). Matches CalendarRevenueCard:287. If it were $state, the RAF effect would re-fire whenever it's written (recursive trigger from inside the effect), causing infinite loop.
