## D-15 Pipeline Strategy Decision

### The actual gap

Live-evidence: `/api/forecast?granularity=day` ships rows with `min(target_date) = today + 1` (i.e., `run_date + 1`). The Python pipeline (`pred_dates_for_grain` at `scripts/forecast/grain_helpers.py:64-65`) generates dates **strictly after `run_date`**:

```python
if granularity == 'day':
    return [run_date + timedelta(days=i + 1) for i in range(horizon)]
```

So when `run_date = 2026-05-03` and `last_actual_date = 2026-04-27`, the pipeline writes forecast rows starting at 2026-05-04. The MV has zero rows for Apr 28-29 (the gap the friend-owner saw) AND zero rows from `window_start = 2026-04-20` through Apr 27. The Python pipeline NEVER writes target_dates ≤ run_date.

### Recommendation: **Option B**

**Rationale:**

1. **Minimal diff.** One file edit (`scripts/forecast/grain_helpers.py`) — `pred_dates_for_grain` already takes `run_date`, `granularity`, `horizon` as kwargs. Add `window_start: date | None = None`. When provided, anchor on `window_start`; when None, behave as today. All 5 model fit scripts pass through unchanged.

2. **No Phase 17 preemption.** Option A would write rows with a new `forecast_track` value (or duplicate `bau` rows with a `is_backtest` flag), forcing Phase 17 BCK-01..08 to negotiate around them. Option B keeps `forecast_track='bau'` semantically clean: "this is what the production model thinks for this date." Phase 17 will introduce `forecast_track='cv_fold_N'` or similar — separate namespace.

3. **Math works because `simulate(anchor='end')` is grain-time-anchored, not date-anchored.** SARIMAX `result.simulate(nsimulations=H, anchor='end')` simulates H steps from the last training observation. The first simulated step's calendar date is whatever you label it as. In the current code, `pred_dates[0] = run_date + 1` and the model "predicts day 1 ahead." Under Option B, `pred_dates[0] = window_start` — the first simulated step is labeled with an earlier calendar date. The model's residual structure is identical; you're just re-labeling the same simulated values to earlier dates. This is **not "running the forecast on past data"** — it's labeling the model's forward-looking prediction with earlier date stamps, which (per D-15) is exactly what the owner asked for: "what would the model have said for these past windowed periods if we'd run it then." [ASSUMED — based on statsmodels `SARIMAXResults.simulate` semantics; verify in implementation]

4. **`horizon_days` GENERATED column accepts negatives.** `target_date - run_date` for backtest rows produces negative integers — no `CHECK` constraint blocks this (verified in `0050_forecast_daily.sql:13`).

5. **Downstream consumers handle naturally:**
   - `last_7_eval.py` filters by `horizon_days > 0` already (no change needed; backtest rows excluded from accuracy logging).
   - `forecast_with_actual_v` already LEFT JOINs to `kpi_daily_mv` on `target_date` — past target_dates will pick up actuals automatically (matching the dashboard's "actuals + forecast on same axis" pattern). No view change needed.
   - `forecast_daily_mv` unique key `(restaurant_id, kpi_name, target_date, model_name, granularity, forecast_track)` is unchanged.
   - API (`/api/forecast +server.ts:95-103`) is unfiltered on `target_date` — past rows ship automatically once written.

6. **Re-running the pipeline tomorrow with a fresh `last_actual` shifts `window_start` slightly forward** (or stays same if data still lags). Upserts overwrite by primary key `(restaurant_id, kpi_name, target_date, model_name, granularity, run_date, forecast_track)`. New `run_date` produces new rows for new target_dates; old run_date rows remain in `forecast_daily` table but `forecast_daily_mv` keeps the latest run via `DISTINCT ON ... ORDER BY ... run_date DESC`. **Operationally clean.**

Add a new helper `window_start_for_grain(last_actual: date, granularity: str) -> date`:

```python
def window_start_for_grain(last_actual: date, granularity: str) -> date:
    """Phase 16.1 D-14: friend-persona window start anchored on last complete period.

    Day:   start_of_week(latest_complete_week_ending_before_or_on(last_actual)) — Mon-anchored.
    Week:  Mon of (latest_complete_week - 4 weeks) — last 5 complete ISO weeks.
    Month: first-of-month(latest_complete_month - 3 months) — last 4 complete months.

    "Latest complete" is relative to last_actual:
      - Day grain: if last_actual is Sun (weekday=6), the week containing it (Mon-Sun)
        is complete. If last_actual is anything Mon-Sat, the previous Mon-Sun week is
        complete. Ex: last_actual=2026-04-27 (Mon, CW18 day 1) → CW18 incomplete →
        latest complete = CW17 (Apr 20-26) → window_start = 2026-04-20.
      - Week grain: same logic, then go back 4 weeks.
      - Month grain: if last_actual is end-of-month, that month is complete; else previous.
    """
    if granularity == 'day':
        # Mon=0..Sun=6. last_complete_week_end = last Sunday on/before last_actual.
        days_after_sunday = (last_actual.weekday() + 1) % 7  # Mon=1, Sun=0
        last_sunday = last_actual - timedelta(days=days_after_sunday)
        # If last_actual IS a Sunday (weekday=6 → days_after_sunday=0), week ending on
        # that Sunday IS the latest complete. Otherwise, latest complete ends on prior Sun.
        if last_actual.weekday() != 6:
            last_sunday = last_actual - timedelta(days=last_actual.weekday() + 1)
        # Week start = Mon = last_sunday - 6 days.
        return last_sunday - timedelta(days=6)
    if granularity == 'week':
        # 5 complete ISO weeks back from latest complete week's Monday.
        # Reuse the day branch to find the Monday of the latest complete week, then -28d.
        day_anchor = window_start_for_grain(last_actual, 'day')
        return day_anchor - timedelta(days=28)
    if granularity == 'month':
        # Latest complete month = last_actual's month if last_actual is end-of-month;
        # else previous month. Then -3 calendar months.
        is_end_of_month = (last_actual + timedelta(days=1)).month != last_actual.month
        if is_end_of_month:
            latest_complete_first = last_actual.replace(day=1)
        else:
            latest_complete_first = (last_actual.replace(day=1) - timedelta(days=1)).replace(day=1)
        return (latest_complete_first - relativedelta(months=3))
    raise ValueError(f'Unknown granularity: {granularity!r}')
```

Modify `pred_dates_for_grain` to accept an optional `window_start`:

def pred_dates_for_grain(
    *,
    run_date: date,
    granularity: str,
    horizon: int,
    window_start: date | None = None,  # NEW: D-15 — anchors past-forecast bucket emission
) -> list:
    """Build native-cadence target_dates from window_start through run_date+horizon.

    When window_start is None (Phase 14/15 behavior), generates [run_date+1, run_date+horizon]
    — the original forward-only range.

    When window_start is provided (Phase 16.1 D-15), generates [window_start, run_date+horizon]
    in the grain's native cadence. Past-forecast rows have target_date < run_date and
    horizon_days < 0 in forecast_daily.
    """
    forward = _forward_pred_dates(run_date, granularity, horizon)  # existing logic
    if window_start is None:
        return forward
    backward = _backward_pred_dates(run_date, granularity, window_start)  # NEW
    return backward + forward

## Window Anchor Math (D-14)

### Worked examples

**Day grain, `last_actual = 2026-04-27` (Monday, CW18 day 1):**
- `last_actual.weekday() = 0` (Mon, not Sun) → `days_after_sunday = 1` → `last_sunday = 2026-04-26` → week start = `2026-04-26 - 6d = 2026-04-20` (Mon). ✓ Matches owner's CW17 expectation (Apr 20-26 is the last complete week).

**Day grain, `last_actual = 2026-04-26` (Sunday):**
- Weekday=6 → it IS a Sunday → CW17 (Apr 20-26) is complete. `last_sunday = 2026-04-26` → window_start = `2026-04-20`. ✓

**Day grain, `last_actual = 2026-05-03` (Sunday):**
- Weekday=6 → CW18 (Apr 27 - May 3) is the latest complete week. window_start = `2026-04-27`.

**Week grain, `last_actual` falls in CW18 partial:**
- Day-grain anchor finds CW17's Monday = `2026-04-20`. Week grain = day_anchor - 28d = `2026-03-23` (Mon of CW13). ✓ Matches owner's "CW13-17 if CW17 is last complete."

**Month grain, `last_actual = 2026-04-27`:**
- April 28 has the same month as April 27 → April is INCOMPLETE → latest complete = March 2026 → first-of-March = `2026-03-01` → minus 3 months = `2025-12-01`. **Note the discrepancy from owner's verbatim spec** (which said "today=May 4 → last complete = April → window_start = 2026-01-01"). Owner's example assumes today=May 4 with `last_actual` already in May; if `last_actual = 2026-04-27` (data lagging into April only), April is NOT complete and the window snaps back to Mar..Dec. **Plan 16.1-04 should ASSUME the data is current enough that latest_complete_month ≥ last_actual.month - 1.** When data lags into mid-month, latest_complete shifts back further — this is correct behavior per the spec's invariant ("relative to last_actual, not today").

## LayerChart 2.0 Behavior Notes

### D-03 chartXDomain widening

**Question:** does LayerChart 2.0 `<Bars>` + `<Spline>` + `<Area>` render correctly when `xDomain` extends past the leftmost bar?

**Answer: YES, verified by reading existing code.**

- `CalendarRevenueCard.svelte:181-192` already sets `chartXDomain = [startAligned, addDays(today, 365)]` where `startAligned` is the bucket-aligned `w.from` and `today + 365d` is FAR past the last bar. Bars render correctly within `[startAligned, today)`; Splines/Areas render in `[today, today+365d]` with no bars; visually the chart has bars on the left and lines on the right. The pattern is **already in production** (Phase 15-12).
- **D-03 widens the LEFT edge** from `bucketRange(w.from)` to `min(bucketRange(w.from), forecast_window_start)`. Symmetric inversion of the existing right-side widening. LayerChart 2.0 `<Bars seriesKey={s.key}>` only emits a bar where `chartData[i]` has data; left of leftmost bar (`forecast_window_start < w.from`) renders nothing for bars but Spline/Area continue to interpolate.
- **Verified Spline behavior:** Spline data is provided via `data={split.past.map((r) => ({ ...r, d: parseISO(r.target_date) }))}`. LayerChart 2.0 Spline (`/techniq/layerchart`) renders only the polyline through provided data points; if `data` covers `[window_start, today]` and `xDomain` is `[window_start, today + 365d]`, the Spline draws cleanly from `window_start` to `today` and stops there. No interpolation past data extents. ✓
- **Verified Area behavior:** Same. Area is data-driven; if data covers `[window_start, today + 365d]` (the full continuous range post-D-15), the band spans the entire xDomain. ✓

**One concrete planner caveat:** when `forecast_window_start == w.from` (current Phase 15-12 case), `chartXDomain = [w.from, today + 365d]` (unchanged). When `forecast_window_start < w.from` (D-14 with default range=30d but window_start back to last complete week ago at narrow ranges), `chartXDomain = [forecast_window_start, today + 365d]` (LEFT-WIDENED). Bars start at `w.from` not `forecast_window_start`; the gap to the left of bars holds only Spline/Area. **This is the desired behavior** but the existing scroll-to-today-at-60% effect (lines 209-263) computes `todayPct = histBuckets / total` — `total` includes only `chartData.length + fcDates.size`. After D-03, the past-forecast bucket count must be ADDED to `total` so the scroll-position ratio stays correct. **Per-plan note for 16.1-01 and 16.1-05.**

## Risks & Open Questions

### Risk 1 — `simulate(anchor='end')` semantics under Option B

**Risk:** SARIMAX `result.simulate(nsimulations=N, anchor='end')` simulates N steps **forward** from the model's last training observation. If we ask for `nsimulations = past_count + future_count = 14 + 372 = 386`, ALL 386 simulated values are forward-looking from `train_end`. We then label the FIRST 14 with past dates (`window_start ... last_actual_date - 1d`).

**Question:** is this semantically defensible? Friend-owner asked: "show me what the model would have said for those past days." The Option B labeling shows: "show me a forward-projection from `train_end`, but call the first 14 outputs 'past' because they happen to land on those calendar dates."

**Reality check:** the model's training cutoff `train_end = last_actual - 7d` (day grain). SARIMAX simulates from `train_end + 1` forward. So the first simulated bucket is `train_end + 1 = last_actual - 6d`, which IS in the past relative to `last_actual`. Under Option B with `window_start = 2026-04-20` and `last_actual = 2026-04-27`, the simulation produces forward steps `train_end+1=2026-04-21 ... train_end+H=2027-04-27` (372 days). The labeled past portion is steps 1-7 (2026-04-21 to 2026-04-27), the labeled future portion is steps 8-386. **THIS WORKS.** The model's "forecast for April 21" IS the value at simulation step 1.

**But: window_start in the day grain is 2026-04-20 (Monday before train_end+1).** That's ONE day BEFORE train_end+1. The first prediction step is for `train_end+1 = 2026-04-21`. So we have NO model output for 2026-04-20 — the window's left edge.

**Resolution:** at day grain, `window_start = 2026-04-20` and `train_end + 1 = 2026-04-21`. The friend-owner's window EXCLUDES the day BEFORE train_end+1. **Reconcile by adjusting `train_end_for_grain` OR letting the first day fall outside the model's prediction range and rendering nothing for that single bucket.**

**Recommendation for planner:** in 16.1-04 Task 1, plan 16.1-04 should:
1. Acknowledge this 1-day edge case in the docstring of `pred_dates_for_grain`.
2. Either (a) skip emitting a row for `window_start` if it equals `train_end` (UI gracefully omits one bucket on the left edge — owner won't notice), OR (b) shift `train_end` forward by one day so `train_end + 1 == window_start + 1` (changes model fitting cutoff slightly — riskier; potentially loses one training observation).
3. **Recommendation: (a) skip the window_start bucket if it precedes train_end+1.** Document as known limitation. The friend-owner's empirical test was about Apr 28-29 (gap in future), not Apr 20 (1-day edge). Acceptable.

### Risk 2 — `prophet_fit.py` may not support `anchor='end'` semantics identically

**Question:** Prophet uses `make_future_dataframe()` rather than `simulate()`. Need to read `prophet_fit.py` to confirm the integration point with `pred_dates_for_grain`.

**Mitigation:** Plan 16.1-04 Task 1 must read all 5 model fit scripts and confirm the integration point. If Prophet's prediction shape can't accept arbitrary past dates, the plan needs to either (a) skip Prophet for past-forecast rows, or (b) use Prophet's `predict()` on a custom dataframe with past dates. Prophet's `predict()` accepts ANY date range, so (b) is feasible. **Action: planner reads prophet_fit.py before locking the implementation.**

### Risk 3 — `last_7_eval.py` accuracy logging excludes backtest rows

**Verified:** `last_7_eval.py:204` (`# horizon_days, evaluation_window, ...`) implies horizon_days filtering. By design, last_7_eval looks at horizon_days=7 evaluation. Past-forecast rows have negative horizon_days. **No accuracy regression.** ✓

### Risk 4 — `forecast_daily_mv` UNIQUE index includes `forecast_track`

**Verified at 0057_forecast_daily_granularity.sql:35-37:** `UNIQUE INDEX forecast_daily_mv_uq ON public.forecast_daily_mv (restaurant_id, kpi_name, target_date, model_name, granularity, forecast_track)`. Option B re-uses `forecast_track='bau'` for both past and future — same key, latest run wins. ✓ no duplicate-key risk.

### Risk 5 — Forecast cards' xDomain auto-scaling vs Calendar* cards' explicit chartXDomain

**Forecast cards** (`RevenueForecastCard.svelte:86-90`) compute `xDomain` from data extremes. After D-15, the leftmost data point IS `window_start`. **No D-03 widening needed on Forecast cards.**

**Calendar* cards** (`CalendarRevenueCard.svelte:181-192`) compute `chartXDomain` from `getWindow()` (user's range filter), NOT from data extremes. So D-03 widening IS needed on Calendar* cards.

**This asymmetry is correct** but planner must NOT confusingly apply D-03 widening to Forecast cards. Plan 16.1-05 explicitly says "leave xDomain unchanged."

### Open Question 1 — Cross-tenant scenario for `window_start_for_grain`

If a future restaurant ships with vastly different freshness profile (e.g., always 2-3 weeks behind), the day-grain window could span 21 days (3 ISO weeks). At v1.3 (single tenant) this is fine. At v1.4+ multi-tenant scale, planner of Phase 17/18 must revisit. **Out of scope for Phase 16.1.**

### Open Question 2 — `naive_dow` model behavior when window_start precedes training data

`naive_dow_fit._fetch_history` reads ALL of `kpi_daily_mv` (limit 10000); training data starts at the restaurant's first business_date. As long as `window_start > first_business_date`, no concern. **For ramen-bones-analytics: first business_date = 2025-06-11 per CLAUDE.md; window_start at day grain ≈ today - 14d. No risk.**
