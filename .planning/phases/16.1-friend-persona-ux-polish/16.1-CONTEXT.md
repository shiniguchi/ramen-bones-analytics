#### Forecast windowing — last complete period anchor (Area E — keystone)

- **D-14 — Window-start definition (owner verbatim spec, locked):**
  - **Day grain:** `window_start = start_of_week(latest_complete_week_ending_before_or_on(last_actual_date))` — Monday-anchored. Example: `last_actual_date = 2026-04-27` (Monday, CW18 day 1) → most-recent-complete-week = CW17 (Apr 20-26) → `window_start = 2026-04-20`. Forecast lines render from Apr 20 through `today + 365d`.
  - **Week grain:** `window_start = start_of_week(latest_complete_week, weekStartsOn=Monday) - 4 weeks` — last 5 complete ISO weeks. Example: `last_actual_date` falls in CW18 (partial) → CW17 is most-recent-complete → `window_start = Monday of (CW17 - 4) = CW13 Monday`. Forecast lines from CW13 through `today + 52w`.
  - **Month grain:** `window_start = start_of_month(latest_complete_month) - 3 months` — last 4 complete calendar months. Example: today=May 4 2026, last complete month = April 2026 → `window_start = 2026-01-01`. Forecast lines from Jan 1 through `today + 12mo`.
  - **"Most-recent-complete-period"** is defined relative to `last_actual_date`, not `today` — if data is current, both align; if data is lagging by weeks, the window anchors on the lag, not on calendar-now.
  - **Cold-start:** if `last_actual_date` is null (zero actuals), researcher picks behavior (recommendation: fall back to `today` as anchor; window = N complete periods ending before today).

- **D-15 — Pipeline ships continuous forecast rows over the windowed range.** The Python forecast pipeline must produce model output rows for **every period** in `[window_start, today + horizon]` for each `(kpi, model, granularity)` combo. No gaps. The MV `forecast_with_actual_v` is the read source; the writer is the pipeline. Implementation choices for the researcher:
  - **Option A — Backtest pre-fill:** add a daily backtest job that computes `yhat` for past periods (rolling-origin). Subset of Phase 17 BCK-01.
  - **Option B — Forecast-from-window-start:** every nightly forecast run anchors `forecast_horizon_start = window_start_per_grain(last_actual_date)`, not `today + 1`. Models project from window_start through `today + horizon` in one shot.
  - **Option C — MV recomputation only:** the MV definition itself joins forward-only forecasts with retrospective backtest forecasts; only the MV needs change.
  - Researcher picks; planner formalizes the migrations + pipeline diff. Acceptance: `/api/forecast?granularity=day` returns a row for every date in `[window_start, today + 365d]` for every visible model — verifiable by `SELECT COUNT(DISTINCT target_date) FROM forecast_with_actual_v WHERE granularity='day'` matching `today - window_start + 365` rows.

- **D-15a — Performance budget for D-15.** Adding past-period forecasts MULTIPLIES the row count in `forecast_with_actual_v`. For day grain: was 376 rows/model, becomes 376 + (today - window_start) ≈ 376 + 14 = 390 rows/model. Negligible. For week+month, even smaller absolute increase. No new performance gate needed for v1.3 scale (1 restaurant). Researcher flags if scale grows.

### Integration Points
- **D-15 + D-16 dependency edge:** D-16 tooltip needs forecast rows for buckets that overlap with bar buckets. The bar buckets span `bucketRange(w.from, w.to, grain)` where `w` comes from `getWindow()`. The forecast rows post-D-15 span `[window_start_per_grain, today + horizon]`. **The intersection** is `[max(w.from, window_start), w.to]`. For day grain with `range=30d` (w.from=2026-04-05) and `window_start=2026-04-20`: tooltip can show forecast values for buckets in [Apr 20, May 4]. Buckets in [Apr 5, Apr 19] have NO forecast row and the tooltip omits the model section (handled by D-16 edge case).
- **D-17 chartXDomain interaction:** forecast cards' xDomain currently spans `[w.from, today + 365d]` (matching Calendar*). With D-14, the past-forecast window can extend BEFORE w.from. D-03 widens it; D-17 needs the same widening.
