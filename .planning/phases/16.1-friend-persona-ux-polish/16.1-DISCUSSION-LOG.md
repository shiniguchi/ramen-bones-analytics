## Past-forecast visual style (Calendar* cards)

### Where should past-forecast styling come from?

| Option | Description | Selected |
|--------|-------------|----------|

**User's choice:** Copy RevenueForecastCard pattern (Recommended).
**Notes:** Researcher must read RevenueForecastCard end-to-end and replicate the past/future layer split.

### How should past-forecast lines visually differ from future-forecast lines?

| Option | Description | Selected |
|--------|-------------|----------|
| Solid faded line for past, dashed for future | Past = solid Spline at ~70% opacity. Future = same line but `stroke-dasharray='4 4'`. Matches backlog's Apple-Health-dotted-expected-line analogy. | ✓ |
| Same stroke style, different opacity | Past at 0.5 opacity, future at 1.0. Subtler difference; risk of looking like a render bug. | |
| Different color: past = zinc-grey total line, future = model color | Past as a single neutral total line; loses per-model continuity in the lag region. | |
| Vertical Rule at last_actual + same-style lines on both sides | One visual divider, same line style. Less elegant than mixed dash/solid. | |

**User's choice:** Solid faded for past, dashed for future (Recommended per backlog).
**Notes:** All visible models render past+future lines (consistent with current behavior).

### Does the chart x-axis domain change?

| Option | Description | Selected |
|--------|-------------|----------|

**User's choice:** Keep current today+365d (Recommended).

### Does the CI band (Area) render in the past-forecast region?

| Option | Description | Selected |

## CampaignUpliftCard hero copy regime tiering

| Option | Description | Selected |
|--------|-------------|----------|

| 2 tiers: <14d / >=14d | Simpler; risk of misleading at 14d when CIs still wide. | |

| You decide during planning | Defer. | |

**User's choice:** 3 tiers per ROADMAP SC#3 (Recommended).

### What drives the day count?

| Option | Description | Selected |
|--------|-------------|----------|
| Use `headline.row.n_days` from /api/campaign-uplift | Already in the payload (Phase 16 D-08). Server-side truth. | ✓ |
| Compute today - campaign_start client-side | Doesn't account for manual-upload lag. | |
| Count distinct dates in `data.daily` array | Lag-aware but adds complexity. | |

**User's choice:** `headline.row.n_days` from API (Recommended).

### Within each tier, should copy branch by CI-overlaps-zero state?

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — each tier has 'overlap zero' and 'doesn't overlap' variants | 5 distinct headlines (since <14d collapses to 1). Matches backlog sketch. | ✓ |
| Tier alone determines copy, ignore CI state | Simpler 3 strings per locale; understates real findings at >=28d confident-negative. | |
| Only branch in the >=28d tier | Compromise: 4 strings per locale; loses early-positive nuance at 14-28d. | |

**User's choice:** Yes — each tier has overlap-zero and doesn't-overlap variants (Recommended).

### Headline tone register — hedged or direct?

| Option | Description | Selected |
|--------|-------------|----------|
| Hedged in early tiers, direct in >=28d | <14d 'too early' — 14-28d 'probably / looks like' — >=28d 'yes/no'. Calibrates confidence to data maturity. | ✓ |
| Always hedged ('probably', 'looks like') | Safer; undersells real findings at >=28d with tight CIs. | |
| Always direct ('yes', 'no', 'no measurable lift') | Clearer; risks being wrong at <14d when CI is wide. | |
| You decide per-string during planning | Planner picks tone per cell of the regime matrix. | |

**User's choice:** Hedged in early tiers, direct in >=28d (Recommended).

## Statistical detail disclosure pattern

### What sits directly under the plain-language hero (the secondary line)?

| Option | Description | Selected |
|--------|-------------|----------|
| Plain-language explanation | e.g., "Best estimate: ~€565 less revenue than expected, but the range covers €3,700 less to €2,300 more — that's normal day-to-day noise." Statistical CI numbers move into disclosure panel. | ✓ |
| Statistical CI numbers (current pattern with plain hero on top) | Hero plain-language; "−€565 (95% CI ...)" stays visible underneath. | |
| Nothing — hero is the whole top section | Cleanest visually but loses the 'best estimate + range' grounding. | |

**User's choice:** Plain-language secondary line (Recommended per backlog).

### How does the owner reveal statistical detail?

| Option | Description | Selected |
|--------|-------------|----------|
| Inline text link with chevron: 'How is this calculated?' › | Tap-to-reveal collapsible inline below card content. Standard mobile pattern. | ✓ |
| Small (i) info icon next to the hero | More compact; less discoverable on mobile. | |
| Bottom sheet / modal | More dramatic, more friction; overkill. | |
| Always-visible smaller text below sparkline | Doesn't solve the 'alphabet soup' problem. | |

**User's choice:** Inline text link with chevron (Recommended).

### What goes inside the disclosure panel?

| Option | Description | Selected |
|--------|-------------|----------|
| Statistical numbers + anticipation buffer + naive disagreement note | All three current bottom-of-card details fold into the panel. | ✓ |
| Statistical numbers only — anticipation buffer + naive note stay always-visible | Bottom-of-card text remains the alphabet soup the owner can't read. | |
| Just a methodology explainer (no specific numbers) | Doesn't really solve the problem. | |

**User's choice:** Statistical numbers + anticipation buffer + naive disagreement (Recommended).

### Default state of the disclosure panel?

| Option | Description | Selected |
|--------|-------------|----------|
| Collapsed by default | Owner sees plain-language card on first visit. Statistical detail one tap away. | ✓ |

**User's choice:** Collapsed by default (Recommended).
