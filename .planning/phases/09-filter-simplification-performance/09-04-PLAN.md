<objective>
Fix a UI reactivity bug that causes the FilterBar and KPI range labels to stay frozen at the SSR snapshot after client-side filter clicks. URL state and dashboardStore computed KPIs already update correctly via `replaceState` + store setters, but everything that reads from `data.filters` stays stale until a full page reload.

Root cause in one sentence: `src/routes/+page.svelte` and `FilterBar`'s filter prop both read from `data.filters` (the frozen SSR snapshot), while click handlers only touch the URL and the dashboardStore's private state — never the object React-shaped components read from.

Fix: make the dashboardStore own a reactive copy of the FiltersState, seed it from SSR on init, extend the existing set* actions to keep it in sync, and rewire `+page.svelte` to read that reactive value instead of `data.filters`. Zero changes to child components (FilterBar, DatePickerPopover, SegmentedToggle, GrainToggle) — they already accept a `filters` prop; making the prop reactive fixes every downstream label/aria-checked/active-preset derivation for free.

Purpose: unblocks UAT Tests 7 and 9 (both flagged `issue`, same root cause per `09-UAT.md` Gaps section with `also_resolves_tests: [9]`). Resolves the cosmetic aria-checked/label staleness noted in Tests 4/5/6 notes as well. After this plan, `/gsd:verify-work 9` should re-run Tests 7 and 9 and flip both to `passed`.

Output:
- `src/lib/dashboardStore.svelte.ts` owns a reactive `_filters` $state object + `getFilters()` getter; `initStore` seeds it; every existing set* setter also updates the matching field(s) on `_filters`.
- `src/routes/+page.svelte` derives `rangeLabel` / `priorLabel` / `handleRangeChange` / the `<FilterBar>` `filters` prop from `getFilters()` (via `$derived`) instead of `data.filters`.
- `tests/unit/dashboardStore.test.ts` gains a small block of tests proving `set*` updates `getFilters()` output.
- Two atomic commits (no Co-authored-by line per `.claude/CLAUDE.md`): store change + page/test change.
