Create `supabase/migrations/0016_insights_table.sql` containing:

1. `CREATE TABLE IF NOT EXISTS public.insights (...)` per D-10 exact schema:
   ```sql
   CREATE TABLE IF NOT EXISTS public.insights (
     id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
     restaurant_id uuid NOT NULL REFERENCES public.restaurants(id),
     business_date date NOT NULL,
     generated_at timestamptz NOT NULL DEFAULT now(),
     headline text NOT NULL,
     body text NOT NULL,
     input_payload jsonb NOT NULL,
     model text NOT NULL,
     fallback_used boolean NOT NULL DEFAULT false,
     CONSTRAINT insights_restaurant_date_key UNIQUE (restaurant_id, business_date)
   );
   ```

2. Lock the raw table per D-16:
   ```sql
   REVOKE ALL ON public.insights FROM authenticated, anon;
   GRANT SELECT, INSERT, UPDATE ON public.insights TO service_role;
   ```

3. Wrapper view with JWT claim filter (matches 0014 shape):
   ```sql
   CREATE OR REPLACE VIEW public.insights_v
     WITH (security_invoker = true) AS
   SELECT id, restaurant_id, business_date, generated_at,
          headline, body, model, fallback_used
   FROM public.insights
   WHERE restaurant_id::text = (auth.jwt()->>'restaurant_id');
   ```
   (Deliberately omit `input_payload` from the wrapper — clients never need the raw payload; it stays in the raw table for audit only.)

4. Grant SELECT on the view:
   ```sql
   GRANT SELECT ON public.insights_v TO authenticated;
   ```

5. Add a RLS policy on the base table (defense in depth — same restaurant_id check as the wrapper):
   ```sql
   ALTER TABLE public.insights ENABLE ROW LEVEL SECURITY;
   CREATE POLICY insights_tenant_read ON public.insights
     FOR SELECT TO authenticated
     USING (restaurant_id::text = (auth.jwt()->>'restaurant_id'));
   ```

Do NOT add a `latest_insight_v` helper — keep surface minimal; the SvelteKit loader (05-04) will `.order('business_date', desc).limit(1)` against `insights_v`.

Create `supabase/migrations/0017_insights_cron.sql` that:

1. Ensures pg_net is available:
   ```sql
   CREATE EXTENSION IF NOT EXISTS pg_net WITH SCHEMA extensions;
   ```

2. Before writing the schedule, VERIFY in `0013_refresh_function_and_cron.sql` that the MV refresh schedule is `0 3 * * *` (UTC). If it is, schedule this job at `15 3 * * *`. If 0013 uses a different time, use `(0013 time) + 15 min` — document the source in a SQL comment.

3. Register the insight job via `cron.schedule` using `pg_net.http_post`. The URL and service-role key come from Vault (do NOT hardcode):
   ```sql
   -- Vault entries expected (created by DevOps step, documented in README Phase 5):
   --   vault.secrets.generate_insight_url         (Edge Function URL)
   --   vault.secrets.generate_insight_bearer      (service_role JWT)
   SELECT cron.schedule(
     'generate-insights',
     '15 3 * * *',
     $$
     SELECT net.http_post(
       url := (SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'generate_insight_url'),
       headers := jsonb_build_object(
         'Content-Type', 'application/json',
         'Authorization', 'Bearer ' || (SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'generate_insight_bearer')
       ),
       body := '{}'::jsonb,
       timeout_milliseconds := 120000
     );
     $$
   );
   ```

4. Make the migration rerunnable — wrap `cron.schedule` in a `DO $$ ... $$` block that unschedules any prior `generate-insights` job first:
   ```sql
   DO $$
   BEGIN
     PERFORM cron.unschedule('generate-insights')
     WHERE EXISTS (SELECT 1 FROM cron.job WHERE jobname = 'generate-insights');
   EXCEPTION WHEN OTHERS THEN NULL;
   END$$;
   ```

5. Add a top-of-file comment citing: "Per 05-CONTEXT D-14 + 05-RESEARCH Pitfall 1 — MV refresh cron is `0 3 * * *` UTC; this job fires 15 min later at `15 3 * * *` UTC (03:15 Europe/Berlin in winter). Decoupled from refresh: either job can fail without blocking the other."

Do NOT create the Vault secrets in this migration (they hold real credentials and must be set via `supabase secrets` / Dashboard out-of-band — document in 05-05 README). The migration tolerates missing secrets gracefully: pg_net will error at cron run time, logged in `cron.job_run_details`, not at migration apply time.
