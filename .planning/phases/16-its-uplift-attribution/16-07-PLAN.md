    Part B — Wrapper views joining campaign_uplift + campaign_calendar.

    Headline wrapper view (per-window aggregates only; **DISTINCT ON dedup keeps only the latest as_of_date per (campaign, model, window_kind)** so the API headline pick is deterministic even though the backing table accumulates one row per nightly run for audit):
    ```sql
    CREATE OR REPLACE VIEW public.campaign_uplift_v AS
    SELECT DISTINCT ON (u.restaurant_id, u.campaign_id, u.model_name, u.window_kind)
      u.restaurant_id,
      u.campaign_id,
      cc.start_date AS campaign_start,
      cc.end_date AS campaign_end,
      cc.name AS campaign_name,
      cc.channel AS campaign_channel,
      u.model_name,
      u.window_kind,
      u.cumulative_uplift_eur,
      u.ci_lower_eur,
      u.ci_upper_eur,
      u.naive_dow_uplift_eur,
      u.n_days,
      u.as_of_date,
      u.computed_at
    FROM public.campaign_uplift u
    INNER JOIN public.campaign_calendar cc
      ON cc.restaurant_id = u.restaurant_id
      AND cc.campaign_id = u.campaign_id
    WHERE u.window_kind IN ('campaign_window', 'cumulative_since_launch')
      AND u.restaurant_id = (auth.jwt()->>'restaurant_id')::uuid
    ORDER BY u.restaurant_id, u.campaign_id, u.model_name, u.window_kind, u.as_of_date DESC;

    GRANT SELECT ON public.campaign_uplift_v TO authenticated;
    COMMENT ON VIEW public.campaign_uplift_v IS 'Phase 16 D-08: read-only campaign_uplift headline rows (per-window aggregates) joined to campaign_calendar; tenant-scoped. DISTINCT ON dedup ensures exactly 1 row per (campaign_id, model_name, window_kind) — the latest as_of_date — even though the backing table accumulates historical rows across nightly runs (each nightly run inserts a fresh aggregate row with as_of_date=run_date for audit/replay). Per-day rows surfaced via campaign_uplift_daily_v.';
    ```

    Why DISTINCT ON: Plan 06's `cumulative_uplift.py` upserts per-window aggregate rows nightly with `as_of_date=run_date`. Because `as_of_date` is part of the PK, each nightly run INSERTs a NEW row instead of overwriting (this is intentional for audit). Without DISTINCT ON dedup, the view would return N rows per (campaign, model, window_kind) after N nights, and `/api/campaign-uplift`'s `find()` headline pick (Plan 08) would be nondeterministic. DISTINCT ON + ORDER BY ... as_of_date DESC surfaces only the most recent aggregate per group. Per-day rows in `campaign_uplift_daily_v` do NOT need this dedup because they each carry `as_of_date=that_day` — the trajectory is naturally one row per day.

    Sister wrapper view for the per-day sparkline trajectory (D-11 shape-of-uplift) — no DISTINCT ON because per-day rows already use `as_of_date=that_day` so each (campaign, model, day) tuple is unique by construction:
    ```sql
    CREATE OR REPLACE VIEW public.campaign_uplift_daily_v AS
    SELECT
      u.restaurant_id,
      u.campaign_id,
      cc.start_date AS campaign_start,
      cc.end_date AS campaign_end,
      cc.name AS campaign_name,
      cc.channel AS campaign_channel,
      u.model_name,
      u.cumulative_uplift_eur,
      u.ci_lower_eur,
      u.ci_upper_eur,
      u.n_days,
      u.as_of_date,
      u.computed_at
    FROM public.campaign_uplift u
    INNER JOIN public.campaign_calendar cc
      ON cc.restaurant_id = u.restaurant_id
      AND cc.campaign_id = u.campaign_id
    WHERE u.window_kind = 'per_day'
      AND u.restaurant_id = (auth.jwt()->>'restaurant_id')::uuid;

    GRANT SELECT ON public.campaign_uplift_daily_v TO authenticated;
    COMMENT ON VIEW public.campaign_uplift_daily_v IS 'Phase 16 D-11: read-only per-day cumulative uplift trajectory for the dashboard sparkline (LayerChart Spline + Area CI band). Tenant-scoped. Same backing table as campaign_uplift_v but filtered to window_kind=per_day rows. No DISTINCT ON because per-day rows are unique by (campaign, model, as_of_date) construction (each day in the window writes once).';
    ```

    Sister-view rationale: clean separation between headline rows (one per campaign×model×window_kind, dedup'd by DISTINCT ON to latest as_of_date) and trajectory rows (N per campaign×model where N = days-in-window, naturally unique). Same backing table, different `window_kind` filter — no duplicate storage. The API filter approach (single view + caller filters) was rejected because consumers must each remember the filter and the headline endpoint should not accidentally fold per-day rows into its aggregate.
    

    Part C — DB CHECK constraint on forecast_daily (primary T-16-05 enforcement per RESEARCH §6):
    ```sql
    -- Phase 16 D-04 (primary mitigation) / RESEARCH §6:
    -- Forbid forecast_track='cf' AND kpi_name='revenue_eur' co-occurrence at the
    -- DB layer. The grep guard (Guard 9, Plan 11) is secondary lint; this CHECK
    -- is mathematically airtight.
    ALTER TABLE public.forecast_daily
      ADD CONSTRAINT forecast_daily_cf_not_raw_revenue
      CHECK (NOT (forecast_track = 'cf' AND kpi_name = 'revenue_eur'));
    ```

    Notes:
    - Comments at top tying back to D-08, RESEARCH §6, T-16-05.
    - Backing table uses `restaurant_id uuid NOT NULL REFERENCES restaurants(id)` (NEVER `tenant_id` — Guard 7).

    - If migration push fails on the CHECK because existing data violates it, the executor must investigate (CF rows from Plan 05 should already source from `revenue_comparable_eur` per Task 4 of Plan 05).
