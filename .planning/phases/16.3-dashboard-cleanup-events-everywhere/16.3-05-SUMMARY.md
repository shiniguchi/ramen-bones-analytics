# Phase 16.3 Plan 05: /api/forecast range_start Query Param Summary

**One-liner:** `/api/forecast` now accepts an optional `?range_start=YYYY-MM-DD` query param that backfills the events feed (4 event tables + campaign_calendar) when supplied; rejects malformed and future dates with HTTP 400 before any DB query; backward compatible.

## What Landed

### Validation gate (lines 78-99 of +server.ts)

```ts
const rangeStartRaw = url.searchParams.get('range_start');
let rangeStart: string | null = null;
if (rangeStartRaw && rangeStartRaw.length > 0) {
  if (!/^\d{4}-\d{2}-\d{2}$/.test(rangeStartRaw)) {
    return json({ error: 'range_start must be YYYY-MM-DD' }, { status: 400, headers: NO_STORE });
  }
  const parsed = parseISO(rangeStartRaw);
  if (Number.isNaN(parsed.getTime())) {
    return json({ error: 'range_start must be YYYY-MM-DD' }, { status: 400, headers: NO_STORE });
  }
  const todayCheck = format(new Date(), 'yyyy-MM-dd');
  if (rangeStartRaw > todayCheck) {
    return json({ error: 'range_start must be ≤ today' }, { status: 400, headers: NO_STORE });
  }
  rangeStart = rangeStartRaw;
}
```

Three failure modes, each returning 400 with `Cache-Control: private, no-store`:
- malformed (e.g. `garbage`, `2025/06/11`) → `range_start must be YYYY-MM-DD`
- parseable-but-invalid (e.g. `2025-13-99`) → `range_start must be YYYY-MM-DD`
- future date (e.g. `2099-01-01`) → `range_start must be ≤ today`

Empty (`?range_start=`) is treated as omitted — no rejection.

### Events lower-bound swap (lines 105-115 of +server.ts)

```ts
const eventsStart = rangeStart ?? todayStr;

const campaignsDefault = format(subDays(today, 90), 'yyyy-MM-dd');
const campaignsStart = rangeStart && rangeStart < campaignsDefault
  ? rangeStart
  : campaignsDefault;
```

4 events queries swap `.gte(...todayStr)` → `.gte(...eventsStart)`:
- `holidays.date`
- `school_holidays.start_date`
- `recurring_events.start_date`
- `transit_alerts.pub_date`

`campaign_calendar.start_date` widens to `range_start` ONLY when the caller asks for an older window than the existing 90-day default (so a card asking for `2024-01-01` gets full campaign coverage; cards asking for the last 30 days continue to receive the 90-day campaign window).

### Untouched (regression-guarded)

- `forecast_with_actual_v` query — windowing controlled by `windowStartForGrain` post-fetch trim (16.1 logic preserved).
- `kpi_daily_v` actuals query — driven by `backtestStart(lastActual, granularity)`, NOT by `range_start`.
- `clampEvents(events, 50)` post-fetch ceiling preserved (CONTEXT.md C-13).
- RLS scope unchanged — all event queries still go through `locals.supabase` (RLS-enforced via `auth.jwt()->>'restaurant_id'`).
- `Cache-Control: private, no-store` header preserved on all 200 + 400 responses.
- No new tables, views, RLS policies, or migrations.
