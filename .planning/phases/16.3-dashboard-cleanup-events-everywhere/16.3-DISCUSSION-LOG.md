## Strip placement relative to chart

| Option | Description | Selected |
|--------|-------------|----------|
| Below bars, above tick labels | Strip sits IMMEDIATELY below the chart canvas, sharing the chart's xScale. Tick labels render below the strip. Date alignment is trivial. | ✓ |
| Below tick labels (separated) | Strip sits below the entire chart including tick labels. Cleaner separation but date alignment requires duplicating the xScale. Adds a 44+px row at the bottom of every wired card — mobile vertical real estate cost. | |
| Above chart, below card title | Strip is the FIRST thing the eye lands on after the title. Pre-eminent visual position. But bars/tick labels then sit further down; on long Calendar cards (with horizontal scroll) the strip may scroll out of view if not sticky. | |
| Overlaid on top edge of chart canvas | Like the existing transit_strike top-bar: badges sit IN the chart canvas at the top, sharing pixels with the bars. Saves vertical space but tap targets clip with tall bars. | |

**User's choice:** Below bars, above tick labels (Recommended)
**Notes:** D-02 in CONTEXT.md. xScale shared with chart for trivial date alignment. 44px height matches tap-target minimum from SC6.

## API range_start derivation

| Option | Description | Selected |
|--------|-------------|----------|
| Caller-supplied (chart x-domain) | Each card passes ?range_start=YYYY-MM-DD to /api/forecast. Card knows its own chart x-domain. Granular per-chart — no over-fetching. | ✓ |

**User's choice:** Caller-supplied chart x-domain (Recommended)
**Notes:** D-05 in CONTEXT.md. Each card knows its own xDomain via existing $derived primitives.
