### NEW Phase 16.3 decisions

#### D-01 — Past-window events feed: extend `/api/forecast`. **LOCKED 2026-05-06 (Recommended).**
Add an optional `?range_start=YYYY-MM-DD` query param to `/api/forecast`. When supplied, the events query backfills events from `range_start` through the existing forecast horizon (`today + 365d`). When omitted, behavior is unchanged (forecast-window only — preserves any other caller's contract). RLS, `clampEvents()`, and `dedupe()` all pass through unchanged — same shared event tables (holidays, school_holidays, recurring_events, campaign_calendar, transit_alerts) per Phase 12 hybrid RLS pattern.

**Why caller-supplied (not server-default-per-grain):** Each card knows its own chart x-domain (`CalendarRevenueCard` extends LEFT to `forecastWindowStart` per 16.1-01; `RepeaterCohortCountCard` extends LEFT to its earliest cohort week/month). Server-default per grain (day=90d / week=26w / month=12m) would over-fetch on tight filter chips and under-fetch on long histories.

**API contract:**
```
GET /api/forecast?kpi=revenue_eur&granularity=day&range_start=2025-06-11
```
- `range_start` optional; absent = today.
- `range_start` must be ≤ today; values > today are rejected (400) — the events array's purpose is to surface past + forecast events, not synthetic future-only.
- The 50-event clamp (`forecastEventClamp.ts`) still applies post-fetch.

#### D-02 — Strip placement: below bars, above tick labels. **LOCKED 2026-05-06 (Recommended).**
EventBadgeStrip renders IMMEDIATELY below the chart canvas, sharing the chart's xScale. Tick labels render BELOW the strip. Date alignment is trivial (same xScale). Mirrors the spirit of `EventMarker.svelte`'s top-bar transit_strike position but inverted to bottom (more readable than top; doesn't compete with bar tops).

**Layout per chart card:**
```
[Card title]
[Chart canvas with bars / Splines / CI Area]
[EventBadgeStrip — fixed height ~44px]
[X-axis tick labels]
[ForecastLegend / ModelAvailabilityDisclosure]
```
Strip height = 44px (matches tap-target minimum from SC6).

#### D-03 — Multi-event badge visual: solid highest-priority color + corner count. **LOCKED 2026-05-06 (Recommended).**
Single-event bucket: solid background = type color (per C-12 palette); 1px white border for legibility on busy charts; rounded-md corners.
Multi-event bucket: solid background = highest-priority event's color (per C-13 priority); white-bordered count badge in top-right corner showing `2`, `3`, `4`, `5+` (literal up to 4, then `+` rollup at 5).
Badge interior: a small icon (or first letter) for single-event clarity is OPTIONAL — planner picks via Chrome MCP at 375×667 (Claude's discretion).

**Rationale:** single-color = instant-recognition for the friend's "I see the campaign came in" use case. Diagonal stripes (rejected option a) become a visual blur at 44px tap targets on mobile.

#### D-04 — `EventMarker.svelte` fate: extract palette + delete. **LOCKED 2026-05-06 (Recommended).**
1. Create `src/lib/eventTypeColors.ts` exporting `EVENT_TYPE_COLORS: Record<EventType, string>` with the 5 hard-coded values currently in `EventMarker.svelte:60-113`. Single source of truth.
2. Optionally re-export `EVENT_PRIORITY` from `forecastEventClamp.ts` via the same module for ergonomics (planner's discretion).
3. Delete `src/lib/components/EventMarker.svelte` — no consumers post-16.3 (the 2 forecast cards being deleted are its only consumers per `grep -rln EventMarker src/`).
4. Spec wording "EventMarker.svelte preserved on disk" interpreted as "the forecast pipeline preserves the visual encoding it provided" — D-04 satisfies that via `eventTypeColors.ts` + the new `EventBadgeStrip` consuming the same palette. Documented inline so future readers don't re-add the file.

#### D-05 — API range param: caller-supplied chart x-domain. **LOCKED 2026-05-06 (Recommended).**
Each card calls `/api/forecast?...&range_start=<earliest_visible_bucket_iso>`. `forecastOverlay.svelte.ts` is the central caller — extend its `ForecastOverlayInputs` type with an optional `rangeStart: () => string | null` getter; pass through to the `/api/forecast` URL.
