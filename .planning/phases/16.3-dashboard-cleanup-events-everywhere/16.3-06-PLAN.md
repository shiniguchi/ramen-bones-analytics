Wave 2 — extend `src/lib/forecastOverlay.svelte.ts` to thread the new `range_start` query param through the existing fetch path AND tighten the events array typing from `unknown[]` to `ForecastEvent[]`. Both changes are CONTEXT.md D-04 + D-05 + Claude's Discretion items.

The factory has 2 existing consumers (`CalendarRevenueCard`, `CalendarCountsCard` per grep). After Plan 16.3-07, `CalendarItemsCard`, `CalendarItemRevenueCard`, `RepeaterCohortCountCard` will optionally consume it as well — Plan 16.3-07 picks the integration option for those 3. By making `rangeStart` optional and additive in this plan, existing 16.1-shipped Calendar* cards continue to work unchanged until Plan 16.3-07 wires them up.

Per CONTEXT.md D-05 contract: "Each card calls `/api/forecast?...&range_start=<earliest_visible_bucket_iso>`. forecastOverlay.svelte.ts is the central caller — extend its `ForecastOverlayInputs` type with an optional `rangeStart: () => string | null` getter; pass through to the `/api/forecast` URL."

Per CONTEXT.md "Claude's Discretion" forecastOverlay typing tightening: "events: unknown[] becomes events: ForecastEvent[] (typed). Implementer follows the existing clampEvents import path; no new type files needed."

Critical reactivity gotcha: the existing `lastFetchedGrain` guard (line 85, 99-100) prevents reactive loops. When we add rangeStart, a caller swapping rangeStart while keeping the same grain MUST trigger a re-fetch. The fix is to widen the cache key from `grain` to `grain|rangeStart`. Without this, range_start works the first time but goes stale on later updates.

Purpose: closes the consumer-side half of D-01 + D-05 (Plan 16.3-05 was the server side). Plan 16.3-07 wires per-card rangeStart getters.
