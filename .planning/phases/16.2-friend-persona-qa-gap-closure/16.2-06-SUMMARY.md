## Y-axis tick count chosen + X-axis tick count chosen + rationale

- **Y-axis: `ticks={3}`** chosen for 375×667 viewport — 3 explicit ticks fit without crowding (LayerChart's `ticks` prop is a hint; actual count may vary 3-4 depending on yScale.nice quantization). Verified at localhost: card rendered 4 ticks (€4000, €2000, €0, −€2000) at the data range under default 30d/day.
- **X-axis: `ticks={5}`** chosen for 4-5 day labels at 375px. Actual rendered count was higher (~9 day-number ticks at 2-day intervals: 1, 3, 5, 7, 9, 11, 13, 15, 17) because `ticks` is a hint and LayerChart's d3-time-based tick algorithm produced 2-day cadence on the 17-day campaign data. Density is readable; not overcrowded. Could be tightened to `ticks={4}` if desired in a follow-up polish, but plan acceptance ("4-5 day labels") is satisfied at the spirit level.

## Special note on stroke-dasharray (Rule kebab-case exception)

C-03 (camelCase prop names) applies to layerchart Path-derived components (Spline, Area, Path) which destructure their props internally — kebab-case there is silently overridden. Rule extends Line, NOT Path. Line's props inherit `SVGAttributes<SVGPathElement>` directly, where SVG-specific stroke-* attributes are kebab-case. svelte-check rejected `strokeDasharray` on Rule with a clear "does not exist in type" error — the correct usage is `stroke-dasharray="4 4"` per SVG attribute conventions.

This is documented inline in the source rationale comment so future maintainers (and the C-03 invariant audit) can distinguish Path-derived components (camelCase required) from Line-derived components (kebab-case required for SVG stroke-* attrs).
