## Conclusion

**Path A — scope-deferred.** DB has only `naive_dow` + `prophet` for week/month at both KPIs. `sarimax`/`ets`/`theta` rows do not exist for non-day granularity. Owner's observation matches DB state.

**Selector behavior:** already data-driven, no code change needed.

`availableModels` is computed from the actual forecast rows in three places:

- `src/lib/components/RevenueForecastCard.svelte:73` — `const availableModels = $derived(Array.from(new Set(rows.map(r => r.model_name))));`
- `src/lib/components/CalendarRevenueCard.svelte:151` — same pattern, derived from forecast rows
- `src/lib/components/CalendarCountsCard.svelte:141` — same pattern, derived from forecast rows

`src/lib/components/ForecastLegend.svelte:43-46` then disables chips not in `availableModels`:

```ts
function isAvailable(modelKey: string): boolean {
  return availableModels.includes(modelKey);
}
```

The disabled chips render at 40% opacity (`opacity-40 cursor-not-allowed`) at `ForecastLegend.svelte:82`. Clicking a disabled chip is a no-op (early-return at `handleClick:55`).

The current treatment ("show all 7 palette chips, grey out the ones DB doesn't have") communicates two facts to the owner: (a) other models exist as a concept, (b) they aren't available at this grain. Hiding them entirely would lose signal (b). No selector edit applied; behavior matches Path A by construction.
