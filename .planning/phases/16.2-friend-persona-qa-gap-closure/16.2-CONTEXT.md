### NEW Phase 16.2 decisions

#### Item 1 — Date-range freeze (perf bug)

- **D-01 — Investigate-first via Chrome MCP performance trace.** First plan task: reproduce the freeze on `localhost:5173` at 375×667 with April + day grain (the documented hang repro). Capture Chrome MCP performance trace + console + network panels. Identify the actual bottleneck before applying any fix. Avoids blind-fix scope creep where we apply 3 candidate fixes and one of them masks a real underlying issue. **G2 LOCKED 2026-05-05.**
- **D-02 — Suspected causes ranked by handoff for the trace to confirm/refute:**
  1. `/api/forecast` row-count blowup — ~385 dates × 5 models × 2 KPIs = ~3850 rows shipped per filter change since 16.1-04 widened forecast windowing.
  2. `replaceState` → `invalidate` re-fetch loop — pre-existing trap documented in memory `feedback_sveltekit_replacestate_invalidate_gotcha.md`. The fix pattern there is `goto({replaceState, invalidateAll})` instead.
  3. `chartXDomain` recompute on every filter change — adds visible across multiple cards, may be quadratic if not memoized.
  4. `computeChartWidth` + scroll-to-today RAF firing in cascade.
- **D-03 — Fix scope determined by trace findings.** Plan structure: Task 1 = trace-and-identify; Task 2 = apply fix(es) the trace points to; Task 3 = verify perf budget at <500ms perceived response.

#### Item 2 — Forecast tooltip (model selection + dot alignment)

- **D-04 — Tooltip row filter:** the `Tooltip.Root` snippet body must filter `data` rows to `selectedModels.has(row.model_name)` AND render every matching row (not just the first). Currently shows only one of N selected models — likely a `bisect-x` collapse bug where the lookup returns a single closest point.
- **D-05 — Per-Spline `Highlight` binding:** each `<Spline>` in the per-model `{#each visibleModels as m}` loop owns its own `<Highlight>` bound to `data={modelData}` (not the chart-level data). Verify dots land on Spline path at 3 sample x-positions via DOM inspection.
- **D-06 — Cross-check Calendar* tooltip while in scope.** D-16 tooltip on `CalendarRevenueCard` + `CalendarCountsCard` may suffer the same model-filter bug — audit during the same plan task; fix if reproducible.

#### Item 3 — Visit-cohort tooltip layout regression

- **D-07 — File scope:** investigator narrows in plan whether the broken card is `RepeaterCohortCountCard.svelte` or `CalendarItemsCard.svelte` (the visit-number variant). Note: handoff mentions `RepeaterCohortRevenueCard.svelte` but only `RepeaterCohortCountCard.svelte` exists; the cards may have been renamed since.
- **D-08 — Layout fix pattern:** `<div class="flex justify-between"><span>{label}</span><span>{value}</span></div>` per row. The bug shows label LEFT / value LEFT-then-RIGHT-spilling — likely a wrapping issue where the model name text wraps and pushes the value to a second line. Fix: `flex-shrink-0` + `whitespace-nowrap` on the value span, OR `truncate` on the label span.
