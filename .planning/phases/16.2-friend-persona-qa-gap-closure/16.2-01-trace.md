## Identified bottleneck

**Dual reactive cascade in `handleRangeChange()` cache-miss path** — `src/routes/+page.svelte` lines ~210–220.

The current cache-miss code:

```ts
setRange(window);            // ← STORE UPDATE 1: writes new dateFrom/dateTo while rawRows is still OLD
await goto(globalThis.window.location.href, {
  replaceState: true,
  invalidateAll: true,
  ...
});                          // ← Server returns new dailyRows; $effect re-runs initStore() with new rawRows
```

This pattern causes the dashboardStore to re-derive its full reactive graph TWICE:

1. **First cascade (LongTask 1, 2053ms):** `setRange(window)` mutates `dateFrom`/`dateTo`. `_filtered = $derived(filterRows(rawRows, ...))` re-runs against the NEW window but the OLD `rawRows` (April dates filtering June-2025 data → mostly empty result). `_bucketed`, `_kpiTotals`, plus every chart card's $derived(getFiltered()) / getBucketed() chain re-runs with the empty result. 10+ cards × LayerChart Spline / Bars / chartXDomain recompute = ~2 seconds main-thread blocking.

2. **~1500ms idle** — server load round-trip.

3. **Second cascade (LongTask 2, 2168ms):** Server returns new `data.dailyRows`/`data.priorDailyRows`. The `$effect(() => initStore({...data}))` at +page.svelte:155-167 fires. `initStore()` mutates `rawRows` (along with re-mutating `dateFrom`/`dateTo`/etc). The full reactive graph re-runs AGAIN with the correct data this time. Same ~2-second cascade work, repeated.

The first cascade is **wasted work** — its output (charts rendering empty April data) flashes on screen for 1 second before the second cascade overwrites it. Removing the pre-goto `setRange()` collapses the dual cascade to a single cascade after fetch returns.

`setRangeId()` is cheap (only updates `_filters` for chip-label derivation; doesn't drive `filterRows`) and can stay.

`/api/forecast` row blowup (D-02 candidate #1) is **not** the bottleneck for range chip changes — forecast endpoint isn't fetched on range change. It may be a separate bottleneck for grain changes (out of scope for this trace; can be a follow-up).

`chartXDomain` recompute (D-02 candidate #3) and `scroll-to-today` RAF (D-02 candidate #4) fire INSIDE each cascade — they aren't the root cause, they're symptoms compounded by running 2× per filter change.

## Fix plan (for Task 2)

**Minimal fix in `src/routes/+page.svelte` `handleRangeChange()`:**

Remove the pre-goto `setRange(window)` line in the cache-miss branch (~line 211). Keep:
- `setRangeId(...)` — cheap, drives chip-label feedback
- Cache-hit branch's `setRange(window)` — needed (no fetch, only client-side update)
- The `$effect` that calls `initStore()` after `goto()` resolves — this owns the post-fetch cascade

After fix: 1 cascade per cache-miss filter change instead of 2. Expected wall-clock drops from ~4-6s to ~1-2s (mostly network round-trip).

If this isn't sufficient to clear the <500ms target, follow-up candidates:
- Investigate why each cascade takes ~2s of main-thread work (likely chartXDomain widening + N forecast cards × 5 models × 365-day horizon Spline path computation)
- Add LazyMount around forecast cards so they don't re-render on every range change (only on grain change which they already self-fetch on)
- Memoize chartXDomain by hashing `[allDates[0], allDates[len-1], grain]`

For Task 2: apply the minimal fix only. Re-measure. Proceed to follow-up only if the trace shows the single remaining cascade is still >500ms blocking.

## Fix applied

**File:** `src/routes/+page.svelte` lines ~204–223 (`handleRangeChange()` cache-miss branch).

**Change:** Removed the pre-goto `setRange(window)` call that mutated `dateFrom`/`dateTo` before the SvelteKit `goto({invalidateAll:true})` SSR round-trip. Added an inline comment naming this trace + the dual-cascade root cause for future maintainers.

**Rationale:** The `$effect(() => initStore({...data}))` already owns the post-fetch store update. Calling `setRange(window)` *before* `goto()` was redundant optimistic work that fired the entire reactive graph against stale `rawRows` (filtering empty April data through old June-2025 rows), then immediately overwrote that work when the fetch returned. The cache-HIT branch's `setRange(window)` is preserved (no fetch, only client-side update). `setRangeId()` is preserved (cheap, drives chip-label feedback during the SSR round-trip).

**Why PARTIAL, not PASS:**
- The dual-cascade root cause identified by Task 1 is fixed. Reactive cascade fires once per filter change instead of twice. Wasted-work elimination is real and significant.
- 1219ms still exceeds the SC-1 truth target of <500ms perceived response. The owner's "freeze" complaint is meaningfully addressed (4.2s → 1.2s blocking, ~71% improvement) but the single remaining cascade is heavy.
- This is a deeper architectural concern: **`initStore()` reassigns ALL reactive state on every cache-miss filter change**, which forces every $derived consumer (10+ chart cards, including LayerChart Spline path generation for forecast cards already mounted via LazyMount) to re-run synchronously on the main thread. Solving it would require bigger surgery (e.g., split `initStore` into `setRawRows` + `setWindow` so consumers that only depend on the window don't re-run on rawRows change; or move heavy chart re-renders to `requestIdleCallback`; or memoize chartXDomain).

**Suspected secondary bottleneck:** `initStore()` mutates ~7 reactive state slots in one synchronous batch. Every $derived(...) in the store + every chart card's $derived chain re-fires. Forecast cards (already-mounted LazyMount Splines × 5 models × ~365-day horizon) are likely the heaviest individual consumers. Calendar* cards (eagerly mounted, not in LazyMount) re-derive on every store update too.

**Recommendation:** Surface this to the user. Two paths forward — accept the 71% improvement + the remaining 1.2s as a v1.4 architectural follow-up (the owner's "freeze" complaint is meaningfully addressed at the wall-clock perception level), or iterate Task 2 with a second-pass fix targeting the single-cascade weight.
