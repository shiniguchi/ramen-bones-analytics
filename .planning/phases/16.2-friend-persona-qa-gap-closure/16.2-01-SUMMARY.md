# 16.2-01 SUMMARY — Date-range freeze perf trace + fix

## Identified bottleneck

**Dual reactive cascade in `handleRangeChange()` cache-miss path** — `src/routes/+page.svelte` lines 204–223. The pre-goto `setRange(window)` call mutated `dateFrom`/`dateTo` while `rawRows` still held the previous range's data, firing every `$derived` consumer (filterRows + 10+ chart cards' chains) once against new window + old data, then again after the SSR fetch when `$effect → initStore()` reset everything.

D-02 candidates ruled out by the trace:
- **#1 `/api/forecast` row blowup** — `/api/forecast` was NOT called on this filter change (forecast cards self-fetch only on grain change). Not relevant for range-chip path.
- **#2 `replaceState`→`invalidate` re-fetch loop** — only 1 fetch per change, the SvelteKit gotcha pattern at line 212 is correctly using `goto({invalidateAll:true})`.
- **#3 `chartXDomain` recompute** — fires inside each cascade, not a root cause.
- **#4 `scroll-to-today` RAF cascade** — fires inside each cascade, not a root cause.

The actual root cause was a NEW candidate the trace surfaced: dual-cascade caused by optimistic store update before `goto()`.

## Fix applied

**File:** `src/routes/+page.svelte` lines 204–223 (`handleRangeChange()` cache-miss branch).

**Change:** Removed the `setRange(window)` call that ran before `await goto({invalidateAll:true})`. The `$effect(() => initStore({...data}))` already owns the post-fetch store update. Cache-HIT branch's `setRange(window)` and both paths' `setRangeId(...)` are preserved.

## Suspected secondary bottleneck (deferred to v1.4)

`initStore()` reassigns ~7 reactive state slots (`rawRows`, `dateFrom`, `dateTo`, `priorFrom`, `priorTo`, `grain`, sales/cash/days filters, `_filters`) in one synchronous batch. Every `$derived` chain in dashboardStore + every chart card's `$derived` chain re-fires on the same tick. Forecast cards (already-mounted LazyMount Splines × 5 models × ~365-day horizon) are likely the heaviest individual consumers. Calendar* cards (eagerly mounted) re-derive on every store update too.

Possible future-pass fixes (not in scope of 16.2-01):
- Split `initStore` into `setRawRows` + `setWindow` so consumers depending only on the window don't re-run on rawRows change.
- Wrap forecast cards' filter-tracking `$derived` in `untrack()` since their primary data flow is `/api/forecast`, not the dashboard store.
- Memoize chartXDomain by hashing `[allDates[0], allDates[len-1], grain]`.
- Move heavy chart re-renders to `requestIdleCallback`.
