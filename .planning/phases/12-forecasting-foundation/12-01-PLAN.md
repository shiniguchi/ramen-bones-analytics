    ```sql
    -- 0039_pipeline_runs_skeleton.sql
    -- Phase 12 FND-09 / D-07: minimal `public.pipeline_runs` skeleton, pulled
    -- forward from Phase 13 so the FND-09 weekly ITS-validity audit cron has

    --

    -- (run_id, step_name, started_at, finished_at, status, row_count, error_msg,

    --
    -- D-08: RLS is INTENTIONALLY ABSENT in this skeleton. The table holds
    -- operational metadata only — no tenant data, no PII, no card_hash, no
    -- PAN. Writes are SERVICE-ROLE ONLY; anon and authenticated have ZERO
    -- access. Phase 13 adds tenant RLS when restaurant_id lands. Do NOT
    -- "fix" this by adding `enable row level security` in Phase 12 — the
    -- skeleton is non-tenant-scoped by design.

    create table if not exists public.pipeline_runs (
      run_id              bigserial primary key,
      step_name           text not null,
      started_at          timestamptz not null default now(),
      finished_at         timestamptz,
      status              text,
      row_count           int,
      error_msg           text,
      commit_sha          text
    );

    -- D-08: skeleton is service-role-only. Browser sessions (anon/authenticated)
    -- cannot read or write. Phase 13's tenant RLS policy lives in a later
    -- migration and is added together with the restaurant_id column.
    revoke all on public.pipeline_runs from anon, authenticated;
    grant select, insert on public.pipeline_runs to service_role;
    ```

    Notes on the formulation:
    - File name `0039_pipeline_runs_skeleton.sql` — `0039` is the next integer after `0038_admin_update_insight_i18n.sql` (last migration in the repo as of 2026-04-28).
    - `create table if not exists` instead of plain `create table` — the migration is idempotent so a manual re-apply does not crash. Aligns with the `do $$ ... end $$` guard pattern used in 0013 for pg_cron job re-registration.
    - All eight columns from D-07 are present in the exact order from CONTEXT.md, with the exact types from CONTEXT.md.
    - No RLS policy. No `enable row level security`. No `restaurant_id` column. Per D-08 + the comment block, this is deliberate.
    - REVOKE first, then GRANT — matches the 0010 cohort_mv idiom (revoke is broader, grant is narrower; order makes the intent obvious).
    - GRANT to `service_role` covers `SELECT, INSERT` only. `UPDATE` and `DELETE` are intentionally NOT granted — `pipeline_runs` is append-only audit trail; any need to mutate a past run is a bug to surface, not silently allow.
