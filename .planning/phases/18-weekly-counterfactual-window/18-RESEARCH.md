## §1 Bootstrap CI methodology for 7-day slice

**Recommendation: Reuse `bootstrap_uplift_ci()` unchanged, called once per completed ISO week.**

### Why the existing function ports cleanly

`scripts/forecast/cumulative_uplift.py:64-126` is a pure-numpy function that takes `actual_values` shape `(N,)` and `yhat_samples_per_day` shape `(N, 200)` and returns `{cumulative_uplift_eur, ci_lower_eur, ci_upper_eur, n_days}`. The function makes **zero assumptions about N** beyond `N == actual_values.shape[0] == paths.shape[0]`. It already runs at N=1 today (the per-day rolling sparkline; see `compute_per_day_uplift_rows` at `cumulative_uplift.py:133-187` which calls it in a loop with seeds `42 + i` for each prefix `[0..i+1)`).

The Phase 16 RESEARCH §1 second-pseudocode-block ("textbook form") is the locked methodology:

```python
# D-08 textbook form, transcribed from cumulative_uplift.py:111-119:
sums = np.empty(n_resamples, dtype=float)
for k in range(n_resamples):
    p = rng.integers(0, P)            # ONE path index per resample, no size= arg
    sums[k] = float((actual_values - paths[:, p]).sum())
ci_lower = float(np.quantile(sums, 0.025))
ci_upper = float(np.quantile(sums, 0.975))
```

This preserves within-path temporal autocorrelation (each draw is one entire stored 200-element path; the 7 sequential draws within a path were generated by Phase 14's native simulation / bootstrap-residual mechanisms which already model dependence). Per-day or block bootstrap is **wrong** here — it would destroy the dependence. Phase 16-RESEARCH already settled this debate; Phase 18 follows the precedent.

### Per-week pseudocode (drop-in for `cumulative_uplift.py`)

```python
# New helper, sibling of compute_per_day_uplift_rows. Call site: main_uplift,
# inside the per-(campaign, model) loop, after compute_uplift_for_window
# returns the loaded actual_values + yhat_samples_per_day arrays.
def compute_iso_week_uplift_rows(
    *,
    restaurant_id: str,
    campaign_id: str,
    model_name: str,
    actual_values: np.ndarray,        # full cumulative-window slice
    yhat_samples_per_day: list,       # full cumulative-window slice
    target_dates: list,               # full cumulative-window slice (date objs)
    today: date,                      # exclude in-progress current week
) -> list:
    """Bucket the cumulative window into ISO weeks (Mon-Sun) and run a fresh
    bootstrap CI for each FULLY COMPLETED past week. Skips:
      - the partial launch week (start of campaign falls mid-week)
      - the in-progress current week (Sun strictly < today)

    Returns one row per completed ISO week with window_kind='iso_week',
    as_of_date = the Sunday of the ISO week (so per-week rows are stable
    across nightly runs and dedup naturally on the PK).
    """
    from datetime import timedelta
    paths = np.asarray(yhat_samples_per_day, dtype=float)
    n = len(target_dates)

    # Bucket day indices by ISO week (year, week_no). isocalendar() returns
    # (year, week, weekday). Mon=1 .. Sun=7 in ISO; Python date.weekday()
    # is Mon=0..Sun=6 — use isocalendar() throughout to stay ISO-canonical.
    buckets: dict[tuple[int, int], list[int]] = {}
    for i, d in enumerate(target_dates):
        iso = d.isocalendar()
        buckets.setdefault((iso.year, iso.week), []).append(i)

    rows: list = []
    for k, ((y, w), idxs) in enumerate(sorted(buckets.items())):
        if len(idxs) < 7:
            # Partial week (leading edge: campaign launches Tue → first ISO
            # week has 6 days; trailing edge: current week is in progress).
            # Both cases excluded per CONTEXT.md leading/trailing-edge rules.
            continue
        week_actuals = actual_values[idxs]
        week_paths = paths[idxs]              # shape (7, 200)
        # Sunday of this ISO week = Monday + 6 days. Identify by weekday=7
        # in iso, but easier: target_dates[idxs[-1]] is the Sunday since we
        # filtered len==7.
        week_end = target_dates[idxs[-1]]
        if week_end >= today:
            # Trailing-edge: skip in-progress week. (Defensive — if pipeline
            # data already excludes today, idxs len would be < 7.)
            continue
        ci = bootstrap_uplift_ci(
            actual_values=week_actuals,
            yhat_samples_per_day=week_paths.tolist(),
            n_resamples=1000,
            # Per-week deterministic seeds that don't collide with the
            # per-day rolling pass (which uses 42 + i, i in [0, n_days)).
            # Offset by 100_000 to keep the namespaces disjoint.
            seed=100_000 + k,
        )
        week_start = target_dates[idxs[0]]
        rows.append({
            "restaurant_id": restaurant_id,
            "campaign_id": campaign_id,
            "model_name": model_name,
            "window_kind": "iso_week",
            "cumulative_uplift_eur": ci["cumulative_uplift_eur"],
            "ci_lower_eur": ci["ci_lower_eur"],
            "ci_upper_eur": ci["ci_upper_eur"],
            "naive_dow_uplift_eur": None,    # cross-check is per-window only
            "n_days": 7,
            "as_of_date": week_end.isoformat(),  # Sunday of ISO week
        })
    return rows
```

### N=7 edge case — is the bootstrap CI meaningful?

**Yes, it is meaningful for this codebase's specific construction**, but the CI **will be much wider** than the 30-day window CI from Phase 16. Two reasons:

1. **The within-path variance for `Σ(actual - path)` over 7 days is √(7/30) ≈ 0.48× the variance over 30 days, but the Σ scale is 0.23× as large.** So the CI is wider as a fraction of the point estimate. This is a feature, not a bug — it tells the friend-owner "one week is too noisy to be sure."
2. **The 200 stored paths sample-saturate fine at N=7.** The bootstrap quantile estimator's source-pool error scales with `1/sqrt(P)` (P=200) regardless of window length N, so 200→1000 path-level resampling produces stable 2.5/97.5 percentiles even for 7-day windows. Phase 16-RESEARCH §1 confirmed this for "windows ≥14 days"; for N=7 the percentile is still well-defined but the absolute CI half-width grows by ~30-50% vs N=14.

**No "n_too_small" gating needed.** The user's CONTEXT.md hero copy already plans for the empty state: when fewer than 1 fully-completed ISO week has passed, the existing UPL-06 maturity-tier "too early" copy reuses (CONTEXT.md §UI / Empty state). For weeks with `n_days = 7` the existing UPL-06 matrix resolves to either `mature_added` / `mature_reduced` (CI cleanly above/below 0) or `mature_no_lift` (CI straddles 0). The existing 7-key i18n set (`uplift_hero_*`) maps cleanly — see `CampaignUpliftCard.svelte:121-146`. **The `n_days < 14 = "early"` and `n_days < 28 = "midweeks"` tiers from Phase 16 are NOT applicable to per-week reads** because each per-week row is a full week regardless of how far the campaign has run; per CONTEXT.md decisions, fully-completed weeks resolve to "mature."

### What `n_resamples` to use?

**Keep 1000.** Same rationale as Phase 16 §1: the bottleneck is the 200-path source pool, not the 1000 resamples. Going to 5000 buys nothing measurable. Going below 1000 risks 2.5/97.5 quantile jitter at the tails (1000 draws × 0.025 = 25 draws define the lower tail — borderline acceptable; 500 would have only ~12 draws which IS too few).

### Edge case: 1 fully-completed ISO week

For a campaign that launched 2026-04-14 (Tue), the first valid bar = ISO week 17 (Apr 20-26 Mon-Sun). As of 2026-05-07 (Wed), that gives **two** completed bars (W17 + W18) and one in-progress (W19). The bootstrap mechanism handles 1-bar and N-bar identically — no special-casing.

**Confidence:** HIGH for the bootstrap math being correct. HIGH for porting `bootstrap_uplift_ci` unchanged. MEDIUM for "1000 resamples sufficient at N=7" — per [scipy.stats.bootstrap](https://docs.scipy.org/doc/scipy/reference/generated/scipy.stats.bootstrap.html) docs the percentile method saturates around 1000-2000 for a 200-source pool regardless of window length.

## §2 ISO week boundary handling

### PostgreSQL — confirmed Mon-based via existing migration

`date_trunc('week', date)` in PostgreSQL **returns the Monday of the ISO week** by definition. Verified by `[CITED: PostgreSQL Date/Time Functions docs]` and confirmed in this codebase by `supabase/migrations/0010_cohort_mv.sql:37`:

```sql
date_trunc('week', fv.first_visit_at at time zone r.timezone)::date as cohort_week,
```

This is the canonical pattern in the codebase. `0034_lift_april_blackout.sql:49` uses the same. ISO 8601 week-numbering year semantics: a year's first ISO week contains January 4. `EXTRACT(isoyear FROM d)` and `EXTRACT(isodow FROM d)` (1=Mon..7=Sun) are available if needed.

**For Phase 18, no SQL ISO-week math is required.** The pipeline computes ISO weeks in Python and writes one `as_of_date` per week (the Sunday). The DB just stores rows. The wrapper view passes them through. SQL ISO-week handling is only needed if a future phase wants to bucket on the SQL side, which this phase doesn't.

### Python — `date.isocalendar()` is canonical

Python 3.9+ `date.isocalendar()` returns a named tuple `(year, week, weekday)` where `weekday: 1=Mon..7=Sun`. **This is what `cumulative_uplift.py` already uses for the `naive_dow` model — see `naive_dow_fit.py:66-67`:**

```python
# isocalendar() returns (year, week, weekday); we use week.
return d.isocalendar()[1]
```

This is the **codebase precedent** — use it in `compute_iso_week_uplift_rows()` for grouping. `pd.Timestamp.isocalendar()` in pandas returns the same shape (year, week, day) as a `Series`-of-frames, but **the existing pipeline uses Python-native `date` objects** (not `pd.Timestamp`) — see `cumulative_uplift.py:194-202` `_to_date()` helper. Stay native for consistency.

### TZ behavior — no TZ work required in this phase

The friend-owner's restaurant is in Berlin. The cumulative_uplift pipeline reads `forecast_with_actual_v` whose `target_date` column is **already TZ-converted to local business date** upstream:

- `kpi_daily_mv` and its derivatives (`kpi_daily_with_comparable_v`) source from `(t.occurred_at at time zone r.timezone)::date as business_date` (`0010_cohort_mv.sql:12`).
- `forecast_daily.target_date` is a `date` (not `timestamptz`) and is written by the pipeline as a Berlin-local business date — `forecast_daily_mv.sql` carries the `granularity='day'` discriminator alone, no TZ column.
- DST: the Berlin business-date conversion absorbs DST internally (Postgres `at time zone` handles the offset shift). Once the per-day forecast row is written with `target_date = '2026-04-14'`, that string IS the local Berlin Monday-Tuesday business day; ISO-week math on top of it inherits the same boundary.

**This means `compute_iso_week_uplift_rows()` can treat `target_dates` as TZ-naive Python `date` objects** and call `.isocalendar()` directly. No `zoneinfo` import, no DST special-case.

### Verification query (for the planner to run as a smoke test)

```sql
-- Confirm: as of 2026-05-07 (Wed), the most recent fully-completed ISO week
-- is Apr 27 (Mon) – May 3 (Sun). Should return: 2026, 18, '2026-04-27', '2026-05-03'.
SELECT
  EXTRACT(isoyear FROM '2026-05-03'::date)::int AS iso_year,
  EXTRACT(week    FROM '2026-05-03'::date)::int AS iso_week,
  date_trunc('week', '2026-05-03'::date)::date  AS iso_week_start,
  (date_trunc('week', '2026-05-03'::date) + interval '6 days')::date AS iso_week_end;
```

**Note on `EXTRACT(week ...)`:** PostgreSQL's `EXTRACT(week ...)` is ISO-8601 week (1-53). `EXTRACT(isoyear ...)` is the ISO week-numbering year (which can differ from the calendar year for late-Dec / early-Jan dates). For the Apr-May 2026 range relevant to this phase, isoyear == year, so this isn't a footgun, but the planner should NOT use `EXTRACT(year ...)` instead — keep `isoyear` to be safe across year-boundary campaigns in future tenants.

**Confidence:** HIGH on all of the above. Verified against `[CITED: PostgreSQL 18 Date/Time Functions docs]` and three places in the codebase using these exact functions.

## §3 LayerChart bars + per-bar CI whiskers + tap interaction

### Bars + onbarclick — verified API

LayerChart 2.x has a built-in `onbarclick` event handler on the `<Bars>` component. **Verified via Context7 [CITED: /techniq/layerchart docs]:**

```svelte
<Bars radius={4} onbarclick={(e, { data }) => alert(data.fruit)} />
```

The callback signature is `(event, { data, series? }) => void` where `data` is the bound row from the chart's `data` array. This is exactly the shape Phase 18 needs:

```svelte
<Bars
  radius={2}
  onbarclick={(e, { data }) => { selectedWeekEnd = data.iso_week_end; }}
/>
```

**Existing codebase usage:** `CalendarRevenueCard.svelte:275-281` already uses `<Bars seriesKey={s.key} rounded={...} radius={4} strokeWidth={1} />` for stacked-bar revenue — no `onbarclick` there yet, but the import path (`from 'layerchart'`) is established. **Mobile touch:** `tooltipContext={{ touchEvents: 'auto' }}` (Phase 15 D-13) ensures bar taps work on phones AND PC trackpads can still scroll the parent container vertically (per `feedback_layerchart_mobile_scroll.md`). Wrap the chart in `class="chart-touch-safe"` like the existing `CampaignUpliftCard.svelte:269` div.

### CI whiskers — compose with `<Rule>`, no built-in error-bar primitive

LayerChart 2.x has **no dedicated error-bar primitive**. The Context7 docs confirm the available primitives include `Rule`, `Bars`, `Spline`, `Area`, `Highlight`, `Axis` — no "ErrorBar". The pattern is to render one `<Rule>` per bar covering `[ci_lower, ci_upper]` at the bar's x-band center, optionally with horizontal end-caps as two short `<Rule>` y-aligned segments.

The cleanest approach is to iterate the data array inside `<Svg>` and emit a `<Rule>` per row. `Rule` accepts `x={...}` and `y={[lo, hi]}` to draw a vertical segment between two y-values at a specified x-position. **Codebase precedent for in-`<Svg>` `<Rule>` usage:** `CampaignUpliftCard.svelte:310` uses `<Rule y={0} class="stroke-zinc-500" stroke-dasharray="4 4" />` for the dashed baseline — same component, just used at a fixed y here.

```svelte
<!-- Pattern for whiskers — emit one Rule per bar inside the Svg block.
     Rule supports both y={value} and y={[lo, hi]} per LayerChart API. -->
{#each weeklyHistory as wk (wk.iso_week_end)}
  <Rule
    x={wk.iso_week_start_date}
    y={[wk.ci_lower_eur, wk.ci_upper_eur]}
    class="stroke-zinc-700"
    stroke-width={1.5}
  />
{/each}
```

Optional whisker end-caps (a small horizontal tick at top and bottom of each whisker) are SVG `<line>` primitives drawn by hand inside the `<Svg>`; this codebase has not done that yet — defer end-caps to a polish pass if the bare vertical-line whiskers don't read clearly at 100px height.

### Color-coding bars conditionally

LayerChart `<Bars>` renders all bars with a single `class` / fill prop — there's no per-row `fill={(d) => ...}` callback in the public 2.x API verified via Context7. Three viable approaches, pick by simplicity:

**Option A (recommended) — Color via class on the data row, applied through `<Bar>` render snippet OR via a CSS-variable-driven approach.** LayerChart 2.x's `<Bars>` iterates and renders individual `<Bar>` marks; you can pass a custom render snippet to override per-bar appearance. Since this codebase hasn't used that snippet before, the simpler default is:

**Option B (simplest) — Render three separate `<Bars>` components, each filtering its slice of the data:**

```svelte
{@const greenBars = weeklyHistory.filter((w) => w.ci_lower_eur > 0)}
{@const redBars = weeklyHistory.filter((w) => w.ci_upper_eur < 0)}
{@const grayBars = weeklyHistory.filter((w) => w.ci_lower_eur <= 0 && w.ci_upper_eur >= 0)}
<Bars data={greenBars} class="fill-emerald-500" radius={2} onbarclick={...} />
<Bars data={redBars}   class="fill-rose-500"    radius={2} onbarclick={...} />
<Bars data={grayBars}  class="fill-zinc-400"    radius={2} onbarclick={...} />
```

This works but bypasses the chart's shared x-band / y-scale — each `<Bars>` element with its own `data` prop overrides the chart context's `data`, which **may misalign x positions** because each bars block computes its band scale from its own subset. **Test before committing to this approach.** If x positions misalign, fall back to Option A.

**Option A's concrete shape (using LayerChart's per-bar render snippet):**

```svelte
<Bars radius={2} onbarclick={(e, { data }) => (selectedWeekEnd = data.iso_week_end)}>
  {#snippet renderBar({ x, y, width, height, data })}
    {@const fillClass =
      data.ci_lower_eur > 0 ? 'fill-emerald-500' :
      data.ci_upper_eur < 0 ? 'fill-rose-500'    :
      'fill-zinc-400'}
    <rect {x} {y} {width} {height} class={fillClass} rx={2} />
  {/snippet}
</Bars>
```

**MEDIUM confidence on the `renderBar` snippet name** — Context7 docs reference `<Bar>` as the inner mark and a snippet override pattern, but the exact prop name (`renderBar`, `bar`, `children`) was not pinned in the snippets I retrieved. The planner should verify with `npx ctx7 docs /techniq/layerchart "Bars children snippet renderBar custom mark"` at plan-execution time and adjust accordingly. **If the snippet API turns out to be unavailable or unstable, fall back to Option B with manual band-scale alignment**, OR simplest fallback: render the bars + colored rectangles via plain SVG `<rect>` inside `<Svg>` referencing chart context's `xScale` / `yScale` directly (the `chartCtx` access pattern is already used in `CalendarRevenueCard.svelte:303-318` via `bind:context={chartCtx}`).

### Selected-bar highlight

For the bar tap to also visually mark the selected week, the simplest pattern is to render an additional `<rect>` overlay on top of the selected bar at `chartCtx.xScale(selectedWeekStart)` with a 2px outline and no fill. The codebase has examples of this overlay-via-`chartCtx` pattern (`CalendarRevenueCard.svelte:303-320` for partial-month text overlays — same shape).

### Mobile considerations

- **280px wide:** matches the existing sparkline canvas. At ~10 weeks of campaign history, that's ~28px per bar — readable, tappable. Beyond ~10 weeks the chart needs horizontal scroll matching the calendar-card pattern (CONTEXT.md UI constraints). **For v1 ship: assume ≤10 weeks of history (campaign launched 2026-04-14, ship date estimated ≤2026-06; ~8 weeks max).** Defer scroll behavior to a follow-up unless the planner sees real cause to bake it in now.
- **Tap target:** WCAG min is 24×24px. At 280×100 with ≤10 bars, each bar is ~28px wide × full chart height = comfortably ≥24×24.
- **`tooltipContext.touchEvents: 'auto'`** is non-negotiable per Phase 15 D-13 / `feedback_layerchart_mobile_scroll.md`. Without it, PC trackpad vertical scroll dies on the chart container.

**Confidence:** HIGH on `onbarclick` API + tooltip touch behavior + Rule-for-whiskers composition. MEDIUM on the per-bar fill snippet name (Option A) — verify at plan-execution time. HIGH that Option B (three filtered `<Bars>` blocks) works as a fallback if Option A's snippet is unstable.

### Recommended addition: `weekly_history` at top level (single-campaign assumption preserved)

// New field — sister to `daily`:
weekly_history: Array<{
  iso_week_start: string;   // ISO date Mon (e.g., '2026-04-27')
  iso_week_end: string;     // ISO date Sun (e.g., '2026-05-03')

**Why top-level (sister to `daily`), not nested in `CampaignBlock`:** The existing `daily` field is top-level even though it's per-campaign-conceptually (the API filters to `campaign_id === headlineCampaignId` at line 124-134). The reason is single-campaign v1 assumption (CONTEXT.md). Mirroring the same shape for `weekly_history` is the path of least resistance and keeps the consumer side trivial:

```ts
// CampaignUpliftCard.svelte derived state:
const weeklyHistory = $derived(data?.weekly_history ?? []);
const selectedWeek = $derived.by(() => {
  if (!data?.weekly_history) return null;
  if (selectedWeekEnd === null) return data.weekly_history.at(-1) ?? null;
  return data.weekly_history.find((w) => w.iso_week_end === selectedWeekEnd) ?? null;
});
```

If/when multi-campaign UX lands (CONTEXT.md deferred), we'd promote `weekly_history` into `CampaignBlock.weekly_history` then. For v1, top-level keeps the API response shape symmetric.

### API code change (3 lines added to `+server.ts`)

```ts
// New: third Promise.all branch reading the iso_week rows.
const [rows, dailyRows, weeklyRows] = await Promise.all([
  fetchAll<UpliftRow>(() => /* existing campaign_uplift_v query */),
  fetchAll<DailyRow>(() => /* existing campaign_uplift_daily_v query */),
  fetchAll<WeeklyRow>(() =>
    locals.supabase

      .eq('model_name', 'sarimax')

// ...later in the response builder, after computing headlineCampaignId:
const weekly_history = headlineCampaignId
  ? weeklyRows
      .filter((w) => w.campaign_id === headlineCampaignId)

        const sun = parseISO(w.as_of_date);
        const mon = subDays(sun, 6);
        return {
          iso_week_start: format(mon, 'yyyy-MM-dd'),
          iso_week_end: w.as_of_date,
          model_name: w.model_name,
          point_eur: w.cumulative_uplift_eur,
          ci_lower_eur: w.ci_lower_eur,
          ci_upper_eur: w.ci_upper_eur,

        };
      })

```

`subDays` from `date-fns` (already imported elsewhere in the codebase). `as_of_date` IS the Sunday end (per the pipeline's write contract); the Monday is `Sun − 6d` deterministically.

### Back-compat preserved per CONTEXT.md

CONTEXT.md API section line 47-48 explicitly preserves `cumulative_deviation_eur` / `ci_lower_eur` / `ci_upper_eur` top-level fields as unused-but-populated. The existing API code at `+server.ts:140` already does this. **No removals in this phase.** A future cleanup phase removes them.

## §5 Migration + RLS extension pattern

### Single-line CHECK constraint extension

Migration 0064 (`supabase/migrations/0064_campaign_uplift_v.sql:48`) defines:

```sql
window_kind text NOT NULL CHECK (window_kind IN ('campaign_window', 'cumulative_since_launch', 'per_day')),
```

The Phase 18 migration must extend that allow-list. The PostgreSQL pattern is:

```sql
-- 0067_campaign_uplift_iso_week.sql (planner picks slot — 0066 is taken)
ALTER TABLE public.campaign_uplift DROP CONSTRAINT campaign_uplift_window_kind_check;
ALTER TABLE public.campaign_uplift
  ADD CONSTRAINT campaign_uplift_window_kind_check
  CHECK (window_kind IN ('campaign_window', 'cumulative_since_launch', 'per_day', 'iso_week'));
```

**Verified-no-side-effects audit:**

- **Indexes:** `campaign_uplift_lookup_idx` is on `(restaurant_id, campaign_id, model_name, window_kind, as_of_date DESC)` (`0064:62-63`). Index does NOT depend on the CHECK constraint values; new `iso_week` rows benefit from the same index. ✓
- **Wrapper views:** `campaign_uplift_v` filters `WHERE u.window_kind IN ('campaign_window', 'cumulative_since_launch')` (`0064:105`). It does NOT include `iso_week` rows by design — those are surfaced via the new sister view. ✓ `campaign_uplift_daily_v` filters `WHERE u.window_kind = 'per_day'` — also unaffected. ✓
- **PK:** `(restaurant_id, campaign_id, model_name, window_kind, as_of_date)` (`0064:56`). New `iso_week` rows write `as_of_date = the Sunday of the ISO week`, which is unique per week per (campaign, model) — no collisions. ✓
- **RLS:** `auth.jwt()->>'restaurant_id'` policy applies to ALL rows in `campaign_uplift` regardless of `window_kind` (`0064:68-69`). New rows inherit the existing policy without modification. ✓
- **`DROP CONSTRAINT IF EXISTS` then `ADD CONSTRAINT`:** PostgreSQL revalidates the new CHECK against existing rows. Existing values are all in `('campaign_window', 'cumulative_since_launch', 'per_day')` which are all in the new allow-list — revalidation passes. The migration is safe on a populated table; no data migration needed. ✓

**Pattern matches** the part-0 of migration 0064 itself, which is the codebase's canonical example of this exact constraint-extension shape (see `0064:29-32` for the prior `forecast_daily_kpi_name_check` extension).

### Sister wrapper view: `campaign_uplift_weekly_v`

CONTEXT.md Q5 asks: extend `campaign_uplift_v` with a DISTINCT ON adjustment, OR add a sister view? **Add a sister view.**

Reasoning:

1. `campaign_uplift_v` uses `DISTINCT ON (...window_kind...)` to dedup per-(campaign, model, window_kind) to the latest `as_of_date` because `campaign_window` and `cumulative_since_launch` rows write a fresh row per nightly run with `as_of_date = run_date`. Multiple nightly runs accumulate; DISTINCT ON keeps only the most recent.
2. **`iso_week` rows do NOT have this multi-run accumulation** — by CONTEXT.md decision (§Pipeline line 27: "as_of_date = the Sunday of the ISO week ... per-week rows are stable across runs and dedup naturally"), each ISO week writes one row per (campaign, model, week). The PK `(restaurant_id, campaign_id, model_name, window_kind, as_of_date)` makes nightly re-runs IDEMPOTENT (the upsert on the same `(campaign, model, 'iso_week', week_sunday)` updates in place; no DISTINCT ON dedup required at the view layer).
3. This is **structurally identical to how `per_day` rows work** — see `campaign_uplift_daily_v` `0064:120-143` which has NO DISTINCT ON because per-day rows are unique by construction. **Mirror that pattern verbatim** for `iso_week`.

Concrete migration template (drop-in, adapted from `0064:120-143`):

```sql
CREATE OR REPLACE VIEW public.campaign_uplift_weekly_v AS
SELECT
  u.restaurant_id,
  u.campaign_id,
  cc.start_date AS campaign_start,
  cc.end_date AS campaign_end,
  cc.name AS campaign_name,
  cc.channel AS campaign_channel,
  u.model_name,
  u.cumulative_uplift_eur,
  u.ci_lower_eur,
  u.ci_upper_eur,
  u.n_days,
  u.as_of_date,             -- = Sunday of the ISO week
  u.computed_at
FROM public.campaign_uplift u
INNER JOIN public.campaign_calendar cc
  ON cc.restaurant_id = u.restaurant_id
  AND cc.campaign_id = u.campaign_id
WHERE u.window_kind = 'iso_week'
  AND u.restaurant_id = (auth.jwt()->>'restaurant_id')::uuid;

GRANT SELECT ON public.campaign_uplift_weekly_v TO authenticated;
COMMENT ON VIEW public.campaign_uplift_weekly_v IS 'Phase 18: read-only per-ISO-week (Mon-Sun) cumulative uplift trajectory for the dashboard bar chart. Tenant-scoped. Same backing table as campaign_uplift_v but filtered to window_kind=iso_week rows. No DISTINCT ON because iso_week rows are unique by (campaign, model, as_of_date=Sunday) by construction (each ISO week writes once and upserts idempotently across nightly runs).';
```

### One migration file per logical unit (codebase invariant)

Per Phase 16 D-15 / `0064` precedent ("One migration per logical unit"), Phase 18 ships ONE migration containing both:
- The CHECK extension (DROP CONSTRAINT + ADD CONSTRAINT)
- The new wrapper view (CREATE VIEW)

Both touch the same table family (`campaign_uplift`); both are atomic and reversible together.

**Confidence:** HIGH. The pattern is mechanical from the existing `0064` template; no novel SQL introduced.

## §7 Risks and gotchas (synthesized)

### R1 — Bootstrap seed collision with the existing per-day pass

`compute_per_day_uplift_rows()` uses seed `42 + i` for `i in [0, n_days)`. For an 80-day campaign that's seeds 42..121. The proposed Phase 18 seed scheme `100_000 + k` (k = ISO week index) keeps the seed namespaces disjoint by construction — no collision risk. **The planner should explicitly document this seed-namespace disjointness in the function docstring** to prevent a future drift introducing collisions (e.g., someone changes per-day seeds to `42 * 1000 + i` and the namespaces overlap).

### R2 — `_load_window_rows` already filters `forecast_track='cf'` and `kpi_name='revenue_comparable_eur'`

The existing `cumulative_uplift.compute_uplift_for_window()` at `cumulative_uplift.py:280-365` does the heavy lifting. Phase 18's `compute_iso_week_uplift_rows()` should be called from `_process_campaign_model` (`cumulative_uplift.py:556-652`) with the SAME `cs["actual_values"]`, `cs["yhat_samples_per_day"]`, `cs["target_dates"]` arrays already loaded for the cumulative-since-launch window. **DO NOT re-query the DB for ISO-week rows** — reuse the cumulative window's loaded arrays and bucket them in Python. This keeps the nightly pipeline at exactly one DB read per (campaign, model) pair and matches how `compute_per_day_uplift_rows` already works (`cumulative_uplift.py:642-650`).

### R3 — Headline-pick deterministic across nightly runs

The card's hero number reads `weekly_history.at(-1)` (most recent ISO week). The `as_of_date` for an `iso_week` row is the Sunday — stable across nightly runs. As long as the API orders `weeklyRows` ascending by `as_of_date` (it does in the proposed §4 query), `at(-1)` is deterministic. **No risk of nondeterministic hero on multi-run days.** Contrast with the existing `campaign_uplift_v` headline pick which DOES need DISTINCT ON dedup because cumulative-since-launch rows accumulate per nightly run.

### R4 — Tap-state preservation across data refreshes

`CampaignUpliftCard.svelte` uses `clientFetch` on mount. If the user taps a bar and then the nightly cron refreshes the data (extremely unlikely on a phone session, but possible on a long-open desktop), `selectedWeekEnd` stays as the user-tapped date. If that date no longer exists in the new `weekly_history` (it always will — past weeks don't disappear), the `find()` returns `undefined` and the hero falls back to `at(-1)`. **Acceptable v1 behavior.** No special-casing needed.

### R5 — `bootstrap_uplift_ci()` requires `EXPECTED_PATHS_PER_ROW = 200` (asserts at line 98-102)

If any forecast_daily row has != 200 paths (e.g., a Phase 14 D-04 storage budget change), the function raises `AssertionError`. Phase 18 is downstream of this — same risk as Phase 16, no new exposure. **No action needed.**

### R6 — DST in pure-Python ISO-week math

Once the pipeline has TZ-converted to Berlin local business date (which it has — see §2), `date.isocalendar()` is a pure-calendar operation and DST transitions don't shift week boundaries. **No DST footgun in this phase.**

### R7 — Single migration, atomic

The CHECK extension (DROP + ADD) and the new view (CREATE OR REPLACE) ship in one migration file. PostgreSQL applies migrations as transactions; if either fails, both roll back. The codebase has no explicit `BEGIN; ... COMMIT;` wrapping in migrations because Supabase's CLI migrator does this implicitly — verify by reading `supabase/migrations/0064_campaign_uplift_v.sql` (no explicit transaction, atomicity via the migrator). Same pattern applies here.

### R8 — Verify the LayerChart per-bar fill snippet API at plan-execution

Highest-uncertainty item in this research. Three viable approaches documented in §3 (Option A snippet, Option B three filtered Bars blocks, Option C manual `<rect>` via chartCtx). The planner should pick at task-creation time after running:

```bash
npx --yes ctx7@latest docs /techniq/layerchart "Bars children snippet renderBar Bar mark custom render"
```

If Option A's snippet API name is confirmed (`renderBar` or `bar` or `children`), use it. If not, fall back to Option C (manual `<rect>` via `bind:context={chartCtx}`) — `CalendarRevenueCard.svelte:303-320` is the codebase precedent for that pattern.
