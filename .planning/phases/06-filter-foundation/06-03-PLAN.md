<objective>
Land the one SSR choke point for filters. Create wrapper view `transactions_filterable_v` (so sales_type + payment_method can be filtered without touching raw `transactions`), then refactor `+page.server.ts` to call `parseFilters(url)` and compose filter-aware queries. Chip-scoped KPI tiles (txCount, avgTicket, chip-window revenue) honor sales_type + payment_method via `.in()`; fixed reference tiles (Today/7d/30d) remain unscoped per UI-SPEC. Load `distinctSalesTypes` and `distinctPaymentMethods` from DISTINCT queries against the new wrapper view at every load (D-14: decoupled from current filter state).

Purpose: FLT-07 demands a single zod-validated SSR pipeline with no dynamic SQL. This plan replaces the ad-hoc `url.searchParams.get()` calls in the loader with `parseFilters()` and wires `.in()`/`.gte()`/`.lte()` chains. Phase 6 scope (D-01) limits filter honoring to what the v1.0 data model supports: KPI tiles fully honor all 4 filters; retention/LTV/frequency/NVR still honor date range only (their views lack sales_type/payment_method dims — Phase 8 adds them). This is documented in the task action so the executor does not over-reach.
Output: Migration 0018 (new wrapper view), loader refactor, loader integration test proving the shape.
</objective>
