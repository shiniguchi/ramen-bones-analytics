<objective>
Stop the deployed dashboard from issuing SSR queries over a 56-year date range. Implement D-01 (chipToRange 'all' resolves to tenant's earliest business_date, with a safe default floor FROM_FLOOR), D-02 (parseFilters soft-clamps from<FROM_FLOOR), and D-05 (fetchAll DEFAULT_MAX_PAGES=50).

Purpose: This is the critical-path production fix. Production returns HTTP 404 on every SSR route because `chipToRange('all')` produces `from=1970-01-01`, which causes `fetchAll` on `transactions_filterable_v` to iterate ~7+ sequential pages — combined with the 10-query fan-out this exceeds Cloudflare Pages Free tier's 50-subrequest / 50ms CPU per request budget, triggers Error 1102, and unpins the Worker. Clamping the range makes that impossible even if the user manually navigates to `from=1970-01-01`, AND — critically — `chipToRange`'s own signature-default is now safe (`FROM_FLOOR='2024-01-01'`), so a future caller that forgets `allStart` can't silently restore the pathological window. Capping `fetchAll` at 50 pages prevents any future regression from silently blowing the subrequest budget.

Output: Three source files with defensive floors (`dateRange.ts`, `filters.ts`, `supabasePagination.ts`), one SSR loader modified to pass `allStart` (`+page.server.ts`), and three test files with new cases proving the clamps behave under pathological input.
</objective>
