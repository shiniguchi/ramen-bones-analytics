## Target User & Narrowest Wedge

**Target user:** the friend running the Berlin ramen restaurant. Single human, single restaurant, in-person observable, ~20 min away.

**Narrowest wedge per office hours:** Sunday Night Text v0. A Sunday-evening **SMS via Twilio** (DE/+49 sender number) sent by **Supabase pg_cron + pg_net** with three lines:
1. Last week's revenue vs prior 4-week median (computed from existing `kpi_daily_mv` or equivalent).
2. The day next week with the lowest historical DOW median (bottom 2 of next 7), flagged with a weather note if OpenWeather forecast shows rain/snow > 5mm. **No SARIMAX/forecast model dependency** — pure DOW-median + weather rule for v0; Phase 14 model can replace this later.
3. One suggested action template ("push the lunch set on Tuesday, rain forecast plus your Tuesdays historically run X% below median").

**Tech specifics (locked, not deferred):**
- **SMS provider:** Twilio SMS to DE +49 number. ~€0.04/msg × 4 Sundays = €0.16. Within $0/mo target. Skipping WhatsApp Business API for v0 because Meta business verification takes days-weeks and breaks the Saturday deadline.
- **Trigger:** Supabase `pg_cron` scheduled at `0 16 * * 0` UTC. Friend receives the message at **17:00 CET in winter / 18:00 CEST in summer** — a deliberate 1-hour drift across DST boundaries. Acceptable for v0 because Sunday-evening early is forgiving; revisit if friend complains. Cron calls a SQL function that uses `pg_net` to POST to Anthropic Haiku for message-text generation, then POST to Twilio for sending. Avoids Edge Function CPU limits. Skeleton:
  ```sql
  -- Wave 0 of Sunday Night Text v0
  select cron.schedule(
    'sunday-night-text',
    '0 16 * * 0',
    $$ select net.http_post(
         url := 'https://api.anthropic.com/v1/messages',
         headers := jsonb_build_object('x-api-key', current_setting('app.anthropic_key'), 'anthropic-version', '2023-06-01', 'content-type', 'application/json'),
         body := jsonb_build_object('model','claude-haiku-4','max_tokens',300,'messages', jsonb_build_array(...)));
       -- then net.http_post to Twilio /Messages.json with the returned text
    $$
  );
  ```
  Production version splits into two SQL functions (one for Haiku, one for Twilio) so retries are cleaner.
- **Weather endpoint:** OpenWeather free-tier 5-day/3-hour forecast (`api.openweathermap.org/data/2.5/forecast?q=Berlin,DE&appid=$KEY&units=metric`). Free tier covers; no ingestion table.
- **Friend's phone:** Stored as Supabase secret `FRIEND_PHONE_E164`, never in source.
- **Reply channel:** Friend replies via SMS to the same Twilio number. Founder reads replies manually (n=4 doesn't justify auto-classification).

This wedge was proposed by the cross-model subagent and matches an option the founder dismissed in Q4. Reviving it as the wedge.

## Approaches Considered

### Approach A: Sunday Night Text wedge first, Phase 13 deferred
Ship the wedge (Twilio SMS + Supabase pg_cron + pg_net + Anthropic Haiku + OpenWeather free tier; pure DOW-median + weather rule, no forecast model), skip Phase 13 build until 2-of-4-Sundays-with-decision proves demand. Effort S, Risk Low, Completeness 5/10. Skips the watch session.

### Approach B: Watch session first, then Phase 13 as scoped
Book the 30-min screen-share, watch silently, find one "if I'd known X" quote. If quote → build Phase 13 as scoped. If no quote → reduce scope. Effort S (30 min), Risk Medium (single signal, observation bias possible), Completeness 7/10.

### Approach C: Both in parallel — watch session AND Sunday Night Text (chosen)
Book watch session + ship Sunday Night Text v0 in the same week. Two independent demand tests in 7 days. Watch captures what friend DOES; Sunday Night Text captures what friend RESPONDS to. After both, decide on Phase 13 with maximum information. Effort M (~2.5 days), Risk Low, Completeness 9/10. Boils the lake.

### Approach D: Reject the diagnostic, build Phase 13 as scoped now
Trust the existing premises P1-P4 are right; ship Phase 13 → 14 → 15 in 3 weeks. Founder did not pick this — the office hours diagnostic moved the founder off "the trajectory IS the wedge."

## Success Criteria

For the office hours assignment (next 7 days):
- [ ] Watch session booked AND completed; one verbatim quote logged in 13-CONTEXT.md or a new 13-OBSERVATION.md.
- [ ] Sunday Night Text v0 deployed and triggered for 1+ Sundays.
- [ ] Friend replies to 1+ Sunday Night Text messages (any reply, even "didn't act on it").

For the Phase 13 greenlight (next 30 days):
- [ ] 2 of 4 Sunday Night Text replies include a specific decision the friend made.
- [ ] Watch-session quote shows pull for at least one of the 5 external-data sources (weather / holidays / school holidays / transit / events).

**Definition of "a specific decision" (locked upfront so the metric is falsifiable):** A reply counts as a real decision IFF it mentions ONE OR MORE of:
- A specific day/date the friend will do/avoid/change something
- A menu item added, cut, or repriced
- A staffing change (more/fewer cooks/servers on a specific shift)
- An inventory order quantity change (cut prep, order more/less of X)
- A promo, discount, or marketing push tied to a specific window
- A hours change (open earlier/later, close earlier/later)

Replies that DO NOT count: "thanks!", emoji-only, "good info", "noted", "I'll think about it", "interesting", any reply that doesn't name a concrete action. Founder reads each reply within 24h and tags it real-decision: yes/no in a flat text log. No auto-classification.

If both pass → Phase 13 build commences with a thesis, not a hope.
If one passes → Phase 13 build commences with reduced scope (drop the unproven sources).
If neither passes → reduce to permanent Sunday Night Text product; defer Phase 13 to v1.4.

## Distribution Plan

The wedge is a Twilio SMS to the friend's existing phone number. No app store, no install, no friend-side onboarding. **Inbound replies** land at a Twilio webhook — for v0, point the webhook at the founder's email (Twilio console > Phone Numbers > Messaging > "A message comes in" → "Forward to email"). Founder reads each inbound SMS within 24h via email and tags real-decision: yes/no in `13-OBSERVATION.md`. No webhook endpoint to build.

The eventual Phase 13/14/15 dashboard layer ships via the existing CF Pages deploy on push to `main` (per `docs/workflow.md` row 13). No new distribution channel needed for the dashboard layer.

**LLM hallucination guard for Sundays 1-2:** Before each of the first two Sunday Night Texts goes out, founder receives a preview SMS to their own phone (Twilio supports a `--dry-run` flag pattern via two recipients in the cron's pg_net call). Founder verifies the headline number ("Tuesday is X% below median") matches the SQL ground truth (one quick query against `kpi_daily_mv`). If mismatch, founder edits/cancels before Twilio sends to friend. Sundays 3+ remove the preview gate once Haiku prompt is calibrated.

**Twilio API failure handling:** pg_cron logs every run to `cron.job_run_details`. After each Sunday 17:00 UTC firing, founder runs `select * from cron.job_run_details where jobname='sunday-night-text' order by start_time desc limit 1` Monday morning. If status != 'succeeded' or pg_net returned non-2xx from Twilio, manually re-trigger or skip that Sunday (and note in success-criteria tracking).
