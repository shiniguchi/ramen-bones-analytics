# Prophet past-projection — Path B revert (16.2 Risk 2 contingency)

**Date:** 2026-05-05
**Phase:** 16.2 (Item 6)
**Source:** Owner persona test 2026-05-05 — `.planning/feedback/16.1-friend-2026-05-05/HANDOFF.md` Item 6 + screenshot `03-prophet-weird-upward-curve-past-not-future.png`

## Lesson

Prophet's `model.predict(future_df)` when `future_df` contains PAST dates (dates ≤ training cutoff) projects the fitted model's stationary trend BACKWARD onto those past dates. This produces exponentially-growing past-yhat values that are NOT backtest-equivalent (i.e., they are NOT what Prophet would have predicted at that historical moment with only the data available up to that moment).

Visually: Prophet's past-Spline shows a smooth exponential climb rather than the noisy, regime-aware actuals — the curve appears as a hallucinated trend rather than a memory of past predictions. The owner's 2026-05-05 screenshot (`03-prophet-weird-upward-curve-past-not-future.png`) shows the artifact unambiguously: an orange line climbing from Oct→Apr that stops at Apr (no future line drawn yet because the chart was misconfigured to render the past curve only).

**The local pytest in 16.1-04 Task 4b only verified the helper math (`window_start_for_grain` returns the right date). It did NOT verify that Prophet's predict on a past-DataFrame produces backtest-equivalent values — that requires a CV harness comparing past-yhat against actuals at the historical moment.** This is exactly the test gap that Phase 17's backtest gate fills.

## Pattern to avoid

Wiring a `window_start` (past date) into Prophet's prediction window without first validating the math via cross-validation:

```python
# AVOID — Path A (16.1-04 Task 4b state, reverted in 16.2-05):
pred_dates = pred_dates_for_grain(
    pred_anchor,
    granularity,
    HORIZON_BY_GRAIN[granularity],
    window_start=window_start_for_grain(last_actual, 'day'),  # ← past date — Risk 2!
    train_end=train_end,
)
future_df = _build_future_df(pred_dates, X_pred)
forecast = model.predict(future_df)  # ← Prophet trend-projects backward
```

Other models (sarimax / naive_dow / ets / theta) do NOT have this issue at the same severity — their `predict` implementations are not stationary-trend-extrapolation-prone in the same way. So `grain_helpers.window_start_for_grain` remains correct for those four; only Prophet's consumption of it is unsafe.

## Pattern to use until Phase 17 backtest gate

**Path B (current default) — skip past-yhat for Prophet only:**

```python
# Prophet only predicts FUTURE; no window_start kwarg.
pred_dates = pred_dates_for_grain(
    pred_anchor,
    granularity,
    HORIZON_BY_GRAIN[granularity],
    # window_start kwarg DROPPED — 16.2 Path B revert (Item 6).
    train_end=train_end,
)
```

The other 4 models (sarimax / naive_dow / ets / theta) keep their windowed past-yhat rows from `grain_helpers.window_start_for_grain` — Path B is Prophet-only.

Calendar* + Forecast cards render Prophet's future-Spline only (no past curve for Prophet); the other 4 models render past + future as before.

## When to revisit (Path A re-evaluation)

Revisit Path A only after **Phase 17 backtest gate** lands a rolling-origin CV harness (see ROADMAP.md Phase 17 SC-1). The harness must validate that Prophet's predict on past dates produces values comparable in error to its at-the-time predictions. If the CV passes, Path A becomes safe. Until then, Path B is the v1.3 default.

The deferred entry mirrors `.planning/phases/16.2-friend-persona-qa-gap-closure/16.2-CONTEXT.md` `<deferred>` section — Phase 17 is the gate that promotes Prophet from forward-only to full-window predictions.

## Test discipline takeaway

Helper-level pytest (does the math return the right *date*?) is not the same as model-level CV (does the *model's prediction* at this date equal what it should have predicted then?). When a CV harness doesn't yet exist, do not wire the helper into a model whose `predict()` implementation is known to extrapolate beyond its training window. Add the safety net first.
