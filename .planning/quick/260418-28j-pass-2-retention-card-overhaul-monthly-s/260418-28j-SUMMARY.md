# quick-260418-28j — Pass 2 Retention Card Overhaul Summary

## One-liner

Monthly cohort retention now reads from a real SQL view (`retention_curve_monthly_v`) instead of a client-side weighted-average re-bucket of weekly rows — period 0 snaps to the definitional 1.0 for all 11 DEV cohorts; x-axis caps at 12 months / 52 weeks; up to 12 cohort lines render via a dedicated palette.

## Executive context

Founder saw monthly retention starting at ~34% in mobile dogfood. Root cause: `CohortRetentionCard.weeklyToMonthly()` collapsed weekly periods 0/1/2 into monthly period 0 and weighted-averaged them, which by construction dilutes period 0 below 1.0. Fix: compute monthly cohorts in SQL (`cohort_mv` → month-grain view), pass both weekly and monthly arrays through the loader, and let the card pick the right series based on grain.

Side goals bundled in the same pass per the plan's 3-bullet scope: (2) cap the chart x-axis so curves don't trail into illegible 200+ week territory, (3) raise the cohort-line cap from 4 to 12 and give those lines a dedicated 12-color palette.

## Task 1 — SQL: retention_curve_monthly_v + test RPC

### Migration body (0027_retention_curve_monthly_v.sql)

Pattern-matched on `0012_leaf_views.sql` retention_curve_v (CTE shape, JWT claim filter, NULL-mask past horizon, `grant select ... to authenticated`, SECURITY DEFINER test RPC). Only semantic divergence is the period expression:

```sql
(extract(year  from age(t.occurred_at, c.first_visit_at)) * 12
 + extract(month from age(t.occurred_at, c.first_visit_at)))::int as period_months
```

Calendar-month diff, which evaluates to 0 whenever `occurred_at` is in the same calendar month as `first_visit_at` (including the first-visit row itself). That's what makes period_months=0 retention_rate = 1.0 by definition.

5-year horizon (`generate_series(0, 60)`) matches weekly's 260-week horizon.
