# Quick Task 260428-5yr — Stop vitest from globbing Deno tests

## Why

`supabase/functions/generate-insight/*.test.ts` use Deno-only imports resolved via `supabase/functions/generate-insight/deno.json` (`std/assert` → `https://deno.land/std@0.224.0/assert/mod.ts`). These are intended to run via `deno test --allow-env --allow-net` (per the deno.json `tasks.test`).

Node-vitest's default test pattern matches them anyway, fails to resolve `std/assert`, and reports 3 failed test files in CI. Excluding `supabase/**` from vitest's discovery removes the noise without changing test coverage — Deno tests stay runnable via `deno test`.

This is one of two issues breaking the Tests workflow on `main` (the other — `Invalid supabaseUrl` from misconfigured `TEST_SUPABASE_URL` secret — is user-action, out of scope here).

## Tasks

### 1. Add `test.exclude` to `vitest.config.ts`

In the `test` config block, set:

```ts
exclude: ['node_modules/**', 'supabase/**', '.svelte-kit/**', 'dist/**']
```

Vitest's default exclude is `node_modules/**, dist/**, .idea/**, .git/**, .cache/**, .{cypress,git}` — adding our own re-states the safe defaults plus our additions. Including `node_modules/**` and `dist/**` keeps coverage equivalent to the default plus exclusions.

**Verify:** `npx vitest run --reporter=dot 2>&1 | grep "supabase/functions/generate-insight"` returns no matches (vitest never tries to load them).

### 2. Sanity-check local test run

Run `npx vitest run --reporter=dot` locally; the 3 Deno test files no longer appear in the output. Other failures (integration tests against missing TEST secrets) may still show locally if `.env` lacks TEST_* — that's expected and orthogonal to this fix.

**Verify:** No `Cannot find package 'std/assert'` errors in the output.

## must_haves

- `vitest.config.ts` has an `exclude` array under `test` that includes `'supabase/**'`.
- Local `npx vitest run` does not attempt to load `supabase/functions/generate-insight/*.test.ts`.

## Out of scope

- Fixing the `Invalid supabaseUrl` integration-test failures (Issue B — secret value).
- Refactoring `tests/helpers/supabase.ts` for fail-soft module-load (Issue C).
- Wiring a `deno test` step into the CI workflow (could be a follow-up).
