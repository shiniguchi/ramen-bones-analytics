Existing pattern for each query (must be preserved):
- Each wrapped in `.then(r => (r.data ?? []) as T[])`
- Each followed by `.catch((e) => { console.error('[name]', e); return [] as T[]; })`
- All six promises resolved together in `Promise.all`

PostgREST pagination contract (from supabase-js docs):
- `.range(from, to)` — inclusive bounds, 0-indexed. e.g., `.range(0, 999)` returns rows 0..999.
- Returns `{ data, error, count }`. When fewer rows are returned than requested,
  the result set is exhausted (partial last page = stop).
- PostgREST enforces `max_rows` on top of .range — so request pageSize MUST be
  `<= max_rows`. Use pageSize = 1000 (Supabase free-tier default, always safe).
- An empty page (`data.length === 0`) also terminates (e.g., rows were an exact multiple of pageSize).

From `tests/unit/pageServerLoader.test.ts` (existing mock pattern to extend):
- `makeSupabase(state)` returns a chainable mock recording `select/gte/lte/in/order/limit/range`
- `chain.then(resolve)` is what `await`-style resolution hits
- `canned` map keyed by table name supplies mocked data
- Add `.range()` to the recorded methods; teach the canned map to slice or return page-scoped data

The helper MUST be transport-agnostic: receive a query builder, apply `.range(from, to)`
in a loop, collect `.data` arrays, stop when a page returns `< pageSize` rows or when
an error is raised. Errors propagate (caller's existing `.catch` converts to `[]`).

    Create `src/lib/supabasePagination.ts` exporting a single function:

    ```typescript
    // Generic pagination over a supabase-js query builder.
    // Why: PostgREST enforces max_rows (typically 1000 on Supabase free tier).
    // A single uncapped .select() silently truncates result sets — the dashboard
    // spent 2026-04-17 showing 1,000-row samples instead of the full 6,896.
    //
    // Usage: await fetchAll(() => supabase.from('x').select('*').gte('d', d0))
    //
    // The builder factory is called once per page — each call must produce a fresh
    // chain so .range() can be appended without re-applying to a previous chain.
    //
    // Stops when: a page returns fewer rows than pageSize OR an empty page.
    // Throws: on any PostgREST error (caller handles via .catch for empty-fallback).
    export async function fetchAll<T>(
      buildQuery: () => PostgrestFilterBuilder<any, any, T[]>,
      pageSize = 1000
    ): Promise<T[]>
    ```

    Loop contract:
    1. `offset = 0`, `all: T[] = []`
    2. `const { data, error } = await buildQuery().range(offset, offset + pageSize - 1)`
    3. If error: throw (wrap in Error with PostgREST message for observability)
    4. If !data || data.length === 0: break
    5. Push data into all. If data.length < pageSize: break (short page = exhausted)
    6. offset += pageSize; loop

    Type the builder parameter loosely — we only need `.range(a, b)` to return a
    thenable resolving to `{ data: T[] | null; error: { message: string } | null }`.
    Import `PostgrestFilterBuilder` from `@supabase/postgrest-js` if the type is
    exported via supabase-js; otherwise declare a minimal structural type inline
    (do NOT use `any` for the return type — T must flow through).

    Safety net: cap max iterations at 1000 (1M rows) with a thrown error if hit —
    prevents runaway loops if a buggy builder always returns full pages.

    Then create `tests/unit/supabasePagination.test.ts` with the 7 tests above.
    Mock the builder by returning an object whose `.range(from, to)` returns a
